# 03_假設型 Persona初稿

## 1. 文件定位

本文件依據 `02_需求洞察整理.md` 產出 2 類假設型 Persona 草稿，用於後續 `04_User Stories` 與 MVP 討論。

使用限制：

- Persona 來自 55 份探索性問卷與人工收斂洞察，尚未經訪談驗證。
- Persona 是開發前討論工具，不代表完整市場分群。
- 匿名是平台預設交流條件，不作為 Persona 類型。

## 2. Persona 總覽

| Persona ID | Persona 名稱 | 一句話描述 | 主要對應洞察 |
|---|---|---|---|
| P-01 | 經驗尋求者 | 遇到人生、職涯或人際問題時，想找有相關經驗的人提供參考，但需要安全、低壓力的交流方式。 | IN-01, IN-02, IN-03, IN-04, IN-05 |
| P-02 | 經驗分享者 | 願意分享自身經驗幫助他人，但需要匿名保護、清楚回覆界線與安全規範。 | IN-01, IN-02, IN-03, IN-05 |

## 3. P-01 經驗尋求者

| 欄位 | 內容 |
|---|---|
| Persona ID | P-01 |
| Persona 名稱 | 經驗尋求者 |
| 一句話描述 | 遇到人生、職涯或人際問題時，想找有相關經驗的人提供參考，但需要安全、低壓力的交流方式。 |
| 代表情境 | 面對職涯選擇、人生方向、人際關係、財務規劃或感情議題時，身邊的人不一定有類似經驗，也不一定方便直接開口詢問。 |
| 核心需求 | 安全可信的交流環境、相關經驗對象、低壓力非即時互動、導覽式輔助入口、清楚簡單的新手流程。 |
| 主要痛點 | 擔心詐騙或個資外洩；怕被評價或說教；不知道對方背景是否可信；不確定怎麼開始提問。 |
| 可能行為 | 先看導覽或範例內容；選擇主題；匿名提出問題；等待非即時回覆；必要時結束交流或檢舉。 |
| 產品機會 | 匿名提問、主題分類、經驗條件選擇、非即時匿名信箱、隱私說明、導覽式入口、檢舉/封鎖。 |
| 證據來源 | F-01, F-03, F-06, F-10, F-11, F-12, F-14, F-15, F-16, T-01, T-02, T-05, IN-01, IN-02, IN-03, IN-04, IN-05 |
| 待驗證假設 | 使用者是否真的願意在平台內提出具體人生問題？導覽/範例內容是否能降低第一次提問門檻？使用者如何判斷對方有「相關經驗」？ |

### 工程師可解釋版本

這類使用者不是單純想聊天，而是遇到具體問題時想找有相關經驗的人參考。產品需要讓他能安全地匿名提問、選主題、理解流程，並避免一開始就承受即時聊天壓力。

### 可支撐的後續 User Stories

- 作為一位經驗尋求者，我希望可以匿名提出問題，以便在不暴露身份的情況下尋求經驗參考。
- 作為一位經驗尋求者，我希望可以選擇問題主題與希望回覆方式，以便找到更符合情境的回覆。
- 作為一位經驗尋求者，我希望第一次使用時可以看到範例或導覽，以便理解平台如何提問與保護安全。
- 作為一位經驗尋求者，我希望可以檢舉或結束不舒服的互動，以便降低匿名交流風險。

## 4. P-02 經驗分享者

| 欄位 | 內容 |
|---|---|
| Persona ID | P-02 |
| Persona 名稱 | 經驗分享者 |
| 一句話描述 | 願意分享自身經驗幫助他人，但需要匿名保護、清楚回覆界線與安全規範。 |
| 代表情境 | 看到他人的問題與自己的經驗相關時，願意提供過來人的看法，但不希望被要求承擔專業責任，也不希望私人資訊被揭露。 |
| 核心需求 | 安全可信的交流環境、相關經驗媒合、低壓力非即時回覆、清楚回覆規範。 |
| 主要痛點 | 擔心匿名不完整；擔心回覆被誤解或被過度依賴；不確定怎麼回才不會變成說教；擔心遇到惡意互動。 |
| 可能行為 | 查看問題摘要；判斷是否與自身經驗相關；匿名回覆；依平台提醒調整語氣；必要時拒絕、結束或檢舉互動。 |
| 產品機會 | 匿名回覆、回覆前友善提醒、經驗非專業建議提示、回覆界線說明、平台內互動限制、檢舉/封鎖。 |
| 證據來源 | F-02, F-03, F-04, F-11, F-12, F-14, F-16, F-18, F-19, T-01, T-03, T-04, T-05, IN-01, IN-02, IN-03, IN-05 |
| 待驗證假設 | 分享者是否願意在非即時信箱中持續回覆？哪些回覆規範能降低說教與錯誤建議？分享者需要哪些界線設定才會安心參與？ |

### 工程師可解釋版本

這類使用者願意幫忙，但不代表願意承擔無限制的責任。產品需要讓他能安全地匿名回覆，並透過回覆規範、非專業建議提示與檢舉機制降低風險。

### 可支撐的後續 User Stories

- 作為一位經驗分享者，我希望可以匿名回覆與我經驗相關的問題，以便在保護身份的情況下幫助他人。
- 作為一位經驗分享者，我希望回覆前能看到友善提醒與回覆界線，以便避免說教或給出不適合的建議。
- 作為一位經驗分享者，我希望可以拒絕或結束不適合的交流，以便維持自己的參與界線。
- 作為一位經驗分享者，我希望平台提醒我的回覆不是專業建議，以便降低被過度依賴或誤解的風險。

## 5. 兩類 Persona 的共同設計原則

| 原則 | 對應洞察 | 說明 |
|---|---|---|
| 匿名是預設條件 | IN-01 | 兩類 Persona 都需要安全匿名，而不是把匿名當作單獨族群。 |
| 先處理安全與信任 | IN-01 | 詐騙、個資、惡意互動、錯誤建議會直接影響參與意願。 |
| 以相關經驗連結雙方 | IN-02 | 尋求者想找有相關經驗的人，分享者也需要判斷問題是否適合自己回覆。 |
| 優先非即時互動 | IN-03 | 非即時信箱能降低壓力，也方便加入平台提醒與審核。 |
| 流程要簡單清楚 | IN-04, IN-05 | 導覽與簡化入口可協助新使用者理解如何提問、回覆與保持安全。 |

## 6. 人工定稿檢查

後續人工定稿時建議確認：

- 兩類 Persona 是否足以支撐後續 6-8 則核心 User Stories。
- 是否需要把「觀望者」作為使用狀態，而不是第三類 Persona。
- Persona 描述是否太像功能清單，是否需要補更多真實使用情境。
- 每張 Persona 的痛點是否都能回到 `F/T` 或 `IN` 證據來源。
- 是否維持「假設型」語氣，不寫成已驗證使用者。
