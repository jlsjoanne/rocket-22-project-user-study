# 02 Affinity 檢核與人工收斂紀錄

## 1. 檢核目的

本文件用來補足 `02_需求洞察整理.md` 的團隊共識紀錄。  
由於目前團隊並非專業 UI/UX 背景，且 `02_需求洞察整理.md` 已完成文字版人工收斂，本階段不強制製作 FigJam Affinity Diagram，而是用 Markdown 表格記錄「證據如何分群、哪些判斷已接受、哪些風險留到後續驗證」。

本文件取代的是 Affinity Diagram 的輕量檢核用途，不取代完整 UX 研究或訪談驗證。

## 4. 人工收斂確認

| 決策項目 | 結論 | 理由 |
|---|---|---|
| 是否保留 5 個正式洞察 | 是 | 5 個洞察已覆蓋安全、經驗媒合、非即時互動、導覽入口與新手流程。 |
| 是否新增第 6 洞察 | 否 | `T-08 平台差異化/配對與回覆誘因` 很重要，但目前更適合列為跨洞察風險與後續驗證問題。 |
| `IN-01` 是否為最高優先 | 是 | 安全審核與檢舉、詐騙/個資風險都直接影響匿名交流是否能開始。 |
| 一對一匿名經驗交流信箱定位 | 主要產品假設 | 問卷支持匿名、相關經驗、安全、非即時需求，但沒有直接證明一對一信箱是唯一最佳形式。 |
| 是否需要完整 FigJam Affinity Diagram | 否 | 目前已有文字版分群、證據追溯與人工收斂紀錄。Markdown 檢核足以作為非 UI/UX 團隊的共識紀錄。 |
| 是否可進入 03 Persona 定稿 | 是 | 02 已提供 Persona 所需的洞察、風險與證據來源。 |

## 5. 進入 03 Persona 的判斷

| Persona | 對應洞察 | 是否足以支撐 | 需在 03 補強的風險 |
|---|---|---|---|
| 經驗尋求者 | IN-01, IN-02, IN-03, IN-04, IN-05 | 是 | 需要補強等待回覆、信箱差異化、配對是否準確、沒人回覆時的替代路徑。 |
| 經驗分享者 | IN-01, IN-02, IN-03, IN-05 | 是 | 需要補強回覆界線、拒絕/略過機制、分享者誘因、非專業建議責任。 |
| 觀望者 | IN-01, IN-04, IN-05 | 暫不新增 Persona | 目前更適合作為使用狀態，放入導覽與新手流程，不獨立成第三 Persona。 |

判斷：`03_假設型 Persona初稿.md` 可繼續作為下一階段人工定稿基礎。後續不需要等待 FigJam 白板完成。

## 6. 待帶入後續階段的問題

| 問題 | 建議帶入階段 | 說明 |
|---|---|---|
| 一對一信箱和論壇/匿名社群的差異化 | 03 Persona, 04 User Stories, 05 MVP | 需要在情境、故事與 MVP 假設中說明平台價值不是單純匿名發文。 |
| 分享者回覆誘因 | 03 Persona, 04 User Stories, 05 MVP | 會影響一對一信箱供給是否穩定。 |
| 沒人回覆或等待過久 | 04 User Stories, 05 MVP | 需要設計等待狀態、候補回覆或導覽內容補位。 |
| 配對不對稱 | 04 User Stories, 05 MVP | 需要用主題、人生經驗與回覆風格降低錯配。 |
| 安全與回覆責任界線 | 03 Persona, 04 User Stories, 05 MVP | 需包含隱私說明、檢舉/封鎖、非專業建議提示與回覆規範。 |

## 7. 結論

`02_需求洞察整理.md` 已完成目前階段必要的人工收斂。  

Affinity Diagram 在本專案目前進度中可作為輕量檢核方法，不需要強制使用 FigJam 模板。若後續老師或團隊要求視覺化，可直接把本文件的分群檢核表轉成 FigJam 便利貼，不需要重新分析。

下一步應進入 `03_假設型 Persona初稿.md` 的人工定稿，確認兩類 Persona 是否符合團隊的一對一匿名經驗交流信箱方向。
