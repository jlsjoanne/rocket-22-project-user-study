## 2. 初步需求主題總覽

| ID | 初版需求主題 | 核心意思 | 可支撐方向 |
|---|---|---|---|
| IN-DRAFT-01 | 安全可信的匿名交流 | 匿名需求存在，但安全與信任是開始互動的前提 | 匿名提問、隱私說明、AI 審核、檢舉/封鎖 |
| IN-DRAFT-02 | 以相關經驗為核心的交流對象 | 使用者重視相關經驗，不只是年齡差異 | 主題選擇、經驗標籤、媒合條件 |
| IN-DRAFT-03 | 低壓力、非即時的互動方式 | 使用者需要能慢慢提問與回覆，降低即時互動壓力 | 匿名信箱、非即時回覆、希望回覆方式 |
| IN-DRAFT-04 | 先瀏覽內容再決定是否互動 | 先看經驗內容是低門檻入口，也能降低冷啟動風險 | 經驗文章、公開經驗牆、內容分類 |
| IN-DRAFT-05 | 清楚簡單的新手流程 | 多互動模式容易混亂，需要明確下一步 | 簡化首頁、新手引導、明確 CTA |
| IN-DRAFT-06 | 回覆品質與互動規範 | 回覆內容需要避免說教、錯誤建議與不當互動 | 回覆規範、友善提醒、非專業建議提示 |

## 3. 初版需求主題證據卡

### IN-DRAFT-01 安全可信的匿名交流

| 欄位 | 內容 |
|---|---|
| 證據來源 | F-01, F-04, F-12, F-14, F-15, T-01, T-02, T-04 |
| 資料看到什麼 | 匿名提問與匿名回覆意願高，但最大阻力集中在詐騙、個資外洩與惡意互動。 |
| 初步需求假設 | 使用者需要在開始匿名交流前，先理解平台如何保護身份、內容與互動安全。 |
| 可支撐 User Stories | 匿名提問、隱私說明、AI 審核、檢舉/封鎖、平台內互動限制 |
| 需人為判讀 | 安全機制是否應列為 MVP 基礎功能，而不是後續加值功能。 |

工程師可解釋版本：
如果平台主打匿名交流，使用者不只在意能不能匿名，也在意匿名後會不會遇到詐騙、被外流或被騷擾。因此安全機制要和匿名功能一起設計。

### IN-DRAFT-02 以相關經驗為核心的交流對象

| 欄位 | 內容 |
|---|---|
| 證據來源 | F-10, F-11, F-12, F-18, F-19, T-03 |
| 資料看到什麼 | 60.0% 填答者認為年齡不重要，重點是對方有相關經驗；主要請教議題集中在職涯、人生方向與人際關係。 |
| 初步需求假設 | 使用者需要找到有相似經驗或能理解情境的人，而不只是找到不同年齡層的人。 |
| 可支撐 User Stories | 主題選擇、經驗標籤、媒合條件、回覆者背景摘要、內容品質提醒 |
| 需人為判讀 | 是否將產品說明重點從「跨世代」調整為「相關經驗／人生階段」。 |

工程師可解釋版本：
媒合邏輯不應只看年齡，而要看使用者遇到什麼問題、想找什麼經驗。這會影響後續資料欄位與媒合條件設計。

### IN-DRAFT-03 低壓力、非即時的互動方式

| 欄位 | 內容 |
|---|---|
| 證據來源 | F-03, F-07, F-12, F-16, T-05 |
| 資料看到什麼 | 一對一匿名信箱有中度支持，匿名文字信箱與不需即時聊天也被明確提及；同時有被說教與被評價風險。 |
| 初步需求假設 | 使用者需要可慢慢組織問題與回覆的互動方式，降低即時聊天帶來的壓力。 |
| 可支撐 User Stories | 匿名信箱、非即時回覆、希望回覆方式、友善提醒、結束交流 |
| 需人為判讀 | 是否將即時配對聊天降級，優先考慮信箱式非即時互動。 |

工程師可解釋版本：
這個平台不一定要做成即時聊天室。非即時信箱可以讓提問者和分享者都有時間思考，也比較容易加入審核與安全提醒。

### IN-DRAFT-04 先瀏覽內容再決定是否互動

| 欄位 | 內容 |
|---|---|
| 證據來源 | F-06, F-08, F-10, F-12, T-03, T-06 |
| 資料看到什麼 | 看經驗文章或故事是最高支持的交流形式；部分填答者也擔心沒人回覆或內容品質不足。 |
| 初步需求假設 | 使用者需要低門檻入口，先透過經驗內容建立信任與理解，再決定是否提問或回覆。 |
| 可支撐 User Stories | 經驗文章/故事瀏覽、公開經驗分享牆、內容分類、收藏、提問前瀏覽 |
| 需人為判讀 | 經驗瀏覽是否作為 MVP 輔助入口，降低冷啟動與直接互動壓力。 |

工程師可解釋版本：
如果一開始沒有人回覆，平台體驗會很弱。先提供可瀏覽內容，可以讓使用者不用立刻互動，也能先感受到平台價值。

### IN-DRAFT-05 清楚簡單的新手流程

| 欄位 | 內容 |
|---|---|
| 證據來源 | F-13, F-12 |
| 資料看到什麼 | 明確下一步提示與簡化首頁分別有 58.2% 與 54.5% 支持。 |
| 初步需求假設 | 使用者需要知道下一步該做什麼，尤其平台若同時包含提問、回覆、信箱與瀏覽，很容易造成入口混亂。 |
| 可支撐 User Stories | 簡化首頁、新手引導、明確 CTA、模式入口分流 |
| 需人為判讀 | 第一版首頁是否只保留 2-3 個主要入口，避免功能同時展開。 |

工程師可解釋版本：
功能入口不能全部攤開。第一版需要清楚告訴使用者可以做什麼，例如先看內容、提出問題、回覆他人。

### IN-DRAFT-06 回覆品質與互動規範

| 欄位 | 內容 |
|---|---|
| 證據來源 | F-16, F-18, F-19, T-03, T-04, T-05 |
| 資料看到什麼 | 使用者擔心被說教、不知道對方背景是否可信，以及收到錯誤或不適合的建議。 |
| 初步需求假設 | 分享者需要被引導如何回覆，尋求者也需要知道回覆只是經驗參考，不是專業建議。 |
| 可支撐 User Stories | 回覆前提醒、經驗非專業建議提示、回覆規範、檢舉不當回覆、希望回覆方式 |
| 需人為判讀 | 是否把回覆規範視為產品內容品質的一部分，而非單純社群守則。 |

工程師可解釋版本：
平台不能只讓使用者自由回覆，還需要在回覆前提示互動規範，降低說教、攻擊或錯誤建議帶來的風險。

## 4. 主題關係整理

| 可合併方向 | 涉及主題 | 說明 |
|---|---|---|
| 平台安全與治理 | IN-DRAFT-01, IN-DRAFT-06 | 安全機制與回覆規範都屬於匿名交流能否被信任的基礎。 |
| 交流媒合與互動方式 | IN-DRAFT-02, IN-DRAFT-03 | 先決定要找什麼樣的人，再決定用信箱、提問或其他方式互動。 |
| 低門檻使用體驗 | IN-DRAFT-04, IN-DRAFT-05 | 先瀏覽內容與簡化入口都能降低新使用者開始使用的門檻。 |

初步建議：

- 若團隊需要 4 個正式洞察，可將 `IN-DRAFT-01` 與 `IN-DRAFT-06` 合併為「安全可信的交流環境」。
- 若團隊需要 5 個正式洞察，可保留 `IN-DRAFT-06`，因為回覆品質會直接影響經驗分享平台的信任感。
- `IN-DRAFT-02` 不建議刪除，因為它會影響後續 Persona、媒合條件與資料欄位設計。

## 5. 下一步人工收斂建議

程式可先產出 6 個初版需求主題，但正式文件仍需團隊人工收斂。建議討論順序如下：

1. 先確認是否接受「匿名交流」作為共同方向。
2. 再確認產品重點是否從「跨世代」調整為「跨人生階段／相關經驗」。
3. 決定要保留 4、5 或 6 個正式洞察。
4. 決定 `IN-DRAFT-01` 安全可信匿名交流是否作為最高優先洞察。
5. 決定「經驗瀏覽」是 MVP 輔助入口，還是後續功能。
6. 將定稿洞察轉成 `03_假設型 Persona` 與 `04_User Stories` 的依據。
