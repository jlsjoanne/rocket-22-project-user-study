# 2 天探索性研究到 User Stories 規劃

## Summary

以 [2天分工及流程.md](/Users/Joanne/Documents/Rocket_Project/Project/2天分工及流程.md)、[工具與協作.md](/Users/Joanne/Documents/Rocket_Project/Project/工具與協作.md) 與 55 份問卷回覆為基礎，產出「探索性研究 → 2 類 Persona → 6-8 則核心 User Stories → MVP 建議」。樣本數不足 100，因此所有結論標註為「假設型需求」，產品規格只作為輔助建議。

核心方向採「一對一匿名經驗交流信箱優先」，但匿名是平台 default 設定，不作為 Persona。主要 Persona 為「經驗尋求者」與「經驗分享者」。

## Key Changes

- Persona：只做 2 類，分別是經驗尋求者、經驗分享者；共同需求包含匿名、安全、低壓力、非即時互動。
- 核心流程：以平台內匿名信箱為主，公開經驗牆與主題內容作為輔助或後續功能。
- 安全邊界：預設平台內互動，不鼓勵交換私人聯絡方式；Must-have 包含檢舉、封鎖、AI 不當內容審核、隱私提示。
- 證據追溯：改用非專業團隊較容易執行的「User Story 證據卡」，不要求每位成員理解完整研究術語。

## Evidence Card Method

每則 User Story 都填一張證據卡，欄位固定如下：

| 欄位 | 內容 |
|---|---|
| Story ID | US-01 |
| Persona | 經驗尋求者／經驗分享者 |
| User Story | 作為一位……我希望……以便…… |
| 支持數據 | 例如匿名提問 61.8%、安全審核 60%、願意匿名提問 70.9% |
| 代表回答／主題 | 例如詐騙、個資外洩、怕被說教、希望慢慢回 |
| 我們的解讀 | 用一句話說明這則故事為什麼成立 |
| 優先級 | Must / Should / Could |
| 驗收條件 | 2-4 條可檢查條件 |
| 風險 | 樣本限制、信任問題、冷啟動、錯誤建議等 |

分工：

- B 補「支持數據」。
- C 補「代表回答／主題」與初版 User Story。
- A 寫「我們的解讀」、確認語氣不過度推論，並整理成報告。

## 2-Day Workflow

Day 1：

- B 完成核心題目的描述統計：匿名提問、匿名回覆、一對一信箱、安全顧慮、提高使用意願、優先功能。
- C 快速整理開放題主題：詐騙、個資、被說教、內容品質、冷啟動、AI 審核。
- A 主持親和圖，把資料收斂成 4-6 個需求主題。
- 全員確認 2 類 Persona 與共同設計原則。

Day 2：

- C 做基本競品分析，聚焦匿名社群、信箱式交流、經驗分享平台。
- A/B/C 共同產出 6-8 則核心 User Stories。
- 每則故事填一張證據卡。
- 用 Must / Should / Could 排 MVP，Must 控制在 5-7 項。
- Notion 彙整研究限制、Persona、User Stories、MVP 與待驗證假設。

## Core Story Criteria

一則 User Story 算「核心」必須符合至少 3 項：

- 支撐一對一匿名經驗交流信箱主流程。
- 對經驗尋求者或經驗分享者有明確價值。
- 問卷中有數據或開放題支持。
- 能降低高風險顧慮，例如詐騙、個資外洩、被說教。
- 可直接轉成 user flow 或 wireframe 畫面。

建議核心故事包含：

- 匿名提出問題。
- 選擇問題主題與希望回覆方式。
- 系統媒合有相關經驗的分享者。
- 分享者在平台內匿名回覆。
- 使用者檢舉、封鎖或結束交流。
- 平台顯示隱私與安全提醒。
- AI 協助檢查不當內容。
- 使用者可先瀏覽經驗內容再決定是否互動。

## User Flow / Wireframe Support

User Stories 後續可直接拆成畫面：

- 提問流程：選主題 → 填問題 → 選匿名/回覆偏好 → 送出。
- 媒合流程：系統提示媒合條件 → 等待回覆 → 收到平台內信箱回覆。
- 分享流程：收到邀請 → 查看問題摘要 → 選擇是否回覆 → 送出匿名經驗。
- 安全流程：檢舉/封鎖 → AI 或平台提示 → 結束互動。
- 瀏覽流程：查看經驗內容 → 收藏/參考 → 決定是否提問。

## Test Plan

- 檢查每則 User Story 是否有證據卡。
- 檢查每張 Persona 至少支撐 2 則故事。
- 檢查 Must-have 是否都能回到問卷數據或開放題主題。
- 檢查報告是否明確寫出「樣本數 55、未訪談、Persona 為假設型」。
- 檢查一對一信箱是否包含安全、檢舉、封鎖、隱私提示，不把安全機制放到後續才做。

## Assumptions

- 團隊固定 3 人，時程固定 2 天。
- 不加入訪談，只做基本競品分析。
- 最終交付以研究到需求為主，產品規格為輔。
- 匿名是平台預設互動設計，不是 Persona 分類。
- 一對一匿名經驗交流信箱是主要 MVP 方向。
