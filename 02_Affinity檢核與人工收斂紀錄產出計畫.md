## 建議文件格式

`02_Affinity檢核與人工收斂紀錄.md` 建議包含：

1. `檢核目的`
   - 說明本文件取代 FigJam Affinity Diagram。
   - 說明這是非 UI/UX 背景團隊使用的文字版分群確認紀錄。

2. `分群檢核表`
   - 欄位：
     - `洞察 ID`
     - `洞察名稱`
     - `放入的 F/T 證據`
     - `團隊是否接受此分群`
     - `需要調整的原因`
     - `最後決定`

3. `人工收斂確認`
   - 確認以下決策：
     - 保留 5 個正式洞察。
     - `IN-01 安全可信的交流環境` 為最高優先。
     - `T-08` 列為風險與待驗證問題，不升級為第 6 洞察。
     - 一對一匿名經驗交流信箱是主要產品假設，不是已驗證結論。

4. `進入 03 Persona 的判斷`
   - 以表格列出：
     - `Persona`
     - `對應洞察`
     - `是否足以支撐`
     - `需在 03 補強的風險`
   - 預期結果：
     - `經驗尋求者` 可進 03。
     - `經驗分享者` 可進 03。
     - `觀望者` 暫列使用狀態，不新增第三 Persona。

5. `待帶入後續階段的問題`
   - 一對一信箱和論壇/匿名社群的差異化。
   - 分享者回覆誘因。
   - 沒人回覆或等待過久。
   - 配對不對稱。
   - 安全與回覆責任界線。

## Test Plan

- 檢查每個 `IN-xx` 至少有 `F-xx` 與 `T-xx` 證據。
- 檢查團隊有明確接受或調整每個洞察分群。
- 檢查文件沒有把一對一信箱寫成已驗證最佳方案。
- 檢查檢核結果能直接支撐 `03_假設型 Persona初稿.md` 的兩類 Persona。
- 檢查後續風險有被保留到 User Stories / MVP 階段。

## Assumptions

- 團隊非專業 UI/UX 背景，因此使用 Markdown 表格比 FigJam 更適合留存與提交。
- Affinity Diagram 在此階段定位為「共識檢核方法」，不是必要視覺交付物。
- 後續若老師或團隊要求視覺化，再把此文件內容轉成 FigJam 即可，不需重新分析。
- 本計畫另存為獨立 Markdown 檔，延續目前專案的決策紀錄規則。
