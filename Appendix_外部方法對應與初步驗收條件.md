# Appendix_外部方法對應與初步驗收條件

## 1. 文件目的

本附錄用來說明 `04_User Stories初稿.md` 中兩個欄位的判讀方式：

- `外部方法對應`
- `初步驗收條件`

目的是讓後續人工審核者在判讀 8 則 User Stories 時，先理解文件構成概念，再決定哪些故事要保留、合併、延後或刪除。

## 2. 外部方法對應是什麼

`外部方法對應` 不是在補充產品需求，而是在說明「這則 User Story 的寫法為什麼合理」。也就是把每則故事對應到外部可靠方法來源，例如 Atlassian、Scrum Alliance、Nielsen Norman Group 的 User Story、Acceptance Criteria 或 UX Story 做法。

它的用途是讓人工審核者知道：

- 這則故事是否有清楚的使用者角色、目標與價值。
- 這則故事是否避免寫成純功能清單。
- 這則故事是否有用具體情境支撐設計理由。
- 這則故事是否保留後續討論空間，而不是直接變成開發規格。

簡單說：`外部方法對應` 是「方法論合理性說明」。它回答的是：

> 這則故事為什麼像一則合格的 User Story？

例如 `US-03` 寫「故事聚焦使用者希望達成的結果，驗收條件避免先指定完整演算法」，意思是：這則故事的重點是「尋求者想找到更相關的回覆」，不是現在就決定系統一定要用什麼媒合演算法。

## 3. 初步驗收條件是什麼

`初步驗收條件` 是把 User Story 轉成比較可檢查的條件。它不是完整規格，也不是最終開發 Definition of Done，而是人工審核時用來判斷「如果要保留這則故事，至少要滿足什麼狀態才算有做到」。

它的用途是讓人工審核者判斷：

- 這則故事是否具體到可以被理解。
- 是否能被後續設計、開發或測試檢查。
- 是否有遺漏必要條件。
- 是否過度承諾目前還沒驗證的事情。

簡單說：`初步驗收條件` 是「最低成功狀態」。它回答的是：

> 如果這則故事成立，使用者至少應該能做到或看到什麼？

例如 `US-02` 的初步驗收條件包含：

- 使用者能匿名建立問題。
- 送出前能看到匿名與隱私提醒。
- 問題不要求真實姓名或私人聯絡方式。
- 送出後能看到後續狀態或下一步。

這些條件不是要求現在立刻開發，而是讓人工審核時能判斷：如果團隊說要做「匿名投遞一對一經驗信箱」，至少不能少掉匿名、隱私提醒、避免個資暴露、送出後狀態這幾件事。

## 4. 人工審核時的使用順序

人工審核 `04_User Stories初稿.md` 時，建議依序查看：

1. 先看 `User Story`：這是不是值得保留的使用者需求？
2. 再看 `支持證據`：它是否有 `F/T/IN` 支撐？
3. 再看 `外部方法對應`：它是否符合 User Story 寫法，而不是功能清單？
4. 最後看 `初步驗收條件`：如果保留這則故事，最低成功狀態是否合理、會不會太大或太細？

## 5. 判讀總結

`外部方法對應` 是用來說服「寫法合理」。

`初步驗收條件` 是用來判斷「做到什麼才算初步成立」。

兩者都不是最終開發規格。它們的作用是協助人工審核者判斷 `04_User Stories初稿.md` 中的故事是否值得保留、是否需要合併、是否該延後到 `05_MVP 建議與待驗證假設`，或是否因證據不足而刪除。
