# 03_假設型 Persona

## 1. 文件定位

本文件由 `03_假設型 Persona初稿.md` 與 `03_假設型Persona人工定稿確認.md` 整合而成，作為進入 `04_User Stories` 前的 Persona 定稿依據。

本文件依據：

- `01_問卷重點整理.md`
- `02_需求洞察整理.md`
- `02_Affinity檢核與人工收斂紀錄.md`
- `03_假設型 Persona初稿.md`
- `03_假設型Persona人工定稿確認.md`

使用限制：

- Persona 來自 65 份探索性問卷與人工收斂洞察，尚未經訪談驗證。
- Persona 是開發前討論工具，不代表完整市場分群。
- 匿名是平台預設交流條件，不作為 Persona 類型。
- 一對一匿名經驗交流信箱是團隊目前選定的主要產品假設，不是問卷已直接證明的唯一最佳形式。
- `觀望者` 不作為第三 Persona，僅作為使用狀態與情境風險，放入安全確認、新手導覽、回覆判斷與信箱等待等描述。

## 2. 人工定稿結論

| 確認項目 | 定稿結論 |
|---|---|
| Persona 數量 | 維持 2 類：經驗尋求者、經驗分享者 |
| 後續 User Stories 支撐性 | 兩類 Persona 可支撐後續 6-8 則核心 User Stories |
| 證據追溯 | 每張 Persona 的痛點需回到 `F/T` 或 `IN` 證據來源 |
| 語氣 | 維持「假設型」語氣，不寫成已驗證使用者 |
| 一對一信箱定位 | 主要產品假設，不是已被問卷完全證明的定案 |
| 觀望者定位 | 使用狀態，不新增第三 Persona |
| 觀望狀態處理 | 放入真實使用情境與新手導覽，不誤解為第三類 Persona |
| 風險保留 | 分享者誘因、配對不對稱、等待過久、安全與回覆責任皆保留為待驗證假設 |
| 真實使用情境 | 每張 Persona 皆補充單段故事格式的真實使用情境 |

## 4. P-01 經驗尋求者

| 欄位 | 內容 |
|---|---|
| Persona ID | P-01 |
| Persona 名稱 | 經驗尋求者 |
| 一句話描述 | 遇到人生、職涯或人際問題時，想找有相關經驗的人提供參考，但需要安全、低壓力且值得等待的交流方式。 |
| 代表情境 | 面對職涯選擇、人生方向、人際關係、財務規劃或感情議題時，身邊的人不一定有類似經驗，也不一定方便直接開口詢問。他可能先觀望平台是否安全、信箱和一般論壇有何不同、是否有人真的能回覆，再決定是否投遞匿名信箱。 |
| 核心需求 | 安全可信的交流環境、相關經驗對象、低壓力非即時互動、導覽式輔助入口、清楚簡單的新手流程。 |
| 主要痛點 | 擔心詐騙或個資外洩；怕被評價或說教；不知道對方背景是否可信；不確定怎麼開始提問；擔心投遞信箱後沒人回或回覆不實用。 |
| 可能行為 | 先看導覽或範例內容；確認匿名與安全說明；選擇主題與希望回覆方式；匿名投遞一對一信箱；等待非即時回覆；必要時結束交流或檢舉。 |
| 產品機會 | 一對一匿名提問信箱、主題分類、經驗條件選擇、回覆風格選擇、等待狀態提示、隱私說明、導覽式入口、檢舉/封鎖。 |
| 證據來源 | F-01, F-03, F-06, F-08, F-10, F-11, F-12, F-13, F-14, F-15, F-16, T-01, T-02, T-05, T-06, T-08, IN-01, IN-02, IN-03, IN-04, IN-05 |
| 待驗證假設 | 使用者是否真的願意在平台內提出具體人生問題？導覽/範例內容是否能降低第一次提問門檻？使用者如何判斷對方有「相關經驗」？一對一信箱是否比公開論壇或匿名社群更有價值？使用者願意等待多久？若沒有人回覆，什麼替代路徑可以維持信任？ |
| 真實使用情境 | 小安最近在考慮是否轉職，但身邊朋友多半沒有類似經驗，也不想讓同事知道自己有離職念頭。他看到平台主打一對一匿名經驗信箱後，會先確認匿名、安全與信箱說明，理解這不是公開發文等留言，再選擇「職涯選擇」主題、描述自己的困惑，並希望收到「分享類似經驗」而不是直接被指導。送出後，他願意等待一段時間，但需要知道目前是否有人接到問題；如果一直沒有人回覆，他希望平台提供等待狀態、相關範例內容或替代建議，而不是讓問題消失。 |

### 工程師可解釋版本

這類使用者不是單純想聊天，而是遇到具體問題時想找有相關經驗的人參考。產品需要讓他能安全地匿名提問、選主題、理解流程，並避免一開始就承受即時聊天壓力。若主功能是一對一信箱，系統還要讓他理解為什麼值得投遞、等待，以及如何處理沒有回覆或回覆不適合的情況。

### 可支撐的後續 User Stories 線索

- 作為一位經驗尋求者，我希望可以匿名投遞一對一經驗信箱，以便在不暴露身份的情況下尋求經驗參考。
- 作為一位經驗尋求者，我希望可以選擇問題主題、相關經驗條件與希望回覆方式，以便提高回覆和自己情境的相關性。
- 作為一位經驗尋求者，我希望第一次使用時可以看到範例或導覽，以便理解信箱和一般論壇發文有什麼不同。
- 作為一位經驗尋求者，我希望可以看到等待狀態或替代建議，以便在沒人回覆時不覺得流程中斷。
- 作為一位經驗尋求者，我希望可以檢舉或結束不舒服的互動，以便降低匿名交流風險。

## 5. P-02 經驗分享者

| 欄位 | 內容 |
|---|---|
| Persona ID | P-02 |
| Persona 名稱 | 經驗分享者 |
| 一句話描述 | 願意分享自身經驗幫助他人，但需要匿名保護、清楚回覆界線、拒絕權與合理參與誘因。 |
| 代表情境 | 看到他人的問題與自己的經驗相關時，願意提供過來人的看法，但不希望被要求承擔專業責任，也不希望私人資訊被揭露。如果問題不符合自己的經驗、太沉重、看起來不真誠，或自己不理解為什麼被分配到這個問題，他需要能略過或拒絕。 |
| 核心需求 | 安全可信的交流環境、相關經驗媒合、低壓力非即時回覆、清楚回覆規範、可拒絕或結束交流的界線。 |
| 主要痛點 | 擔心匿名不完整；擔心回覆被誤解或被過度依賴；不確定怎麼回才不會變成說教；擔心遇到惡意互動；不確定為什麼要持續回覆陌生人的問題。 |
| 可能行為 | 查看問題摘要；判斷是否與自身經驗相關；略過不適合的提問；匿名回覆；依平台提醒調整語氣；必要時拒絕、結束或檢舉互動。 |
| 產品機會 | 匿名回覆、回覆前友善提醒、經驗非專業建議提示、回覆界線說明、拒絕/略過機制、平台內互動限制、檢舉/封鎖、分享者參與誘因。 |
| 證據來源 | F-02, F-03, F-04, F-11, F-12, F-14, F-16, F-17, F-19, T-01, T-03, T-04, T-05, T-08, IN-01, IN-02, IN-03, IN-05 |
| 待驗證假設 | 分享者是否願意在非即時信箱中持續回覆？哪些回覆規範能降低說教與錯誤建議？分享者需要哪些界線設定才會安心參與？什麼誘因會讓分享者願意回覆？分享者是否需要知道對方問題和自己經驗的匹配原因？略過或拒絕是否會降低分享者壓力並提升長期參與意願？ |
| 真實使用情境 | 阿哲曾經歷過轉職低潮，願意分享經驗幫助別人。當他收到一則匿名提問時，他會先看問題摘要、主題與系統標示的匹配原因，判斷自己是否真的有相關經驗。如果問題太沉重、和自己經驗不符，或看起來像不真誠互動，他希望可以略過而不被懲罰。若決定回覆，他希望平台提醒自己以經驗分享為主，不要做專業建議或說教，也不要交換私人聯絡方式；回覆完成後，他也需要知道互動可以在平台內結束，不會被迫持續承擔對方後續決策。 |

### 工程師可解釋版本

這類使用者願意幫忙，但不代表願意承擔無限制的責任。產品需要讓他能安全地匿名回覆，並透過回覆規範、非專業建議提示、拒絕/略過機制與檢舉機制降低風險。若平台主打一對一信箱，分享者端的篩選、界線與參與動機會直接影響供給是否穩定。

### 可支撐的後續 User Stories 線索

- 作為一位經驗分享者，我希望可以收到與我經驗相關的一對一匿名提問，以便在保護身份的情況下幫助他人。
- 作為一位經驗分享者，我希望回覆前能看到友善提醒與回覆界線，以便避免說教或給出不適合的建議。
- 作為一位經驗分享者，我希望可以略過、拒絕或結束不適合的交流，以便維持自己的參與界線。
- 作為一位經驗分享者，我希望平台提醒我的回覆不是專業建議，以便降低被過度依賴或誤解的風險。
- 作為一位經驗分享者，我希望知道為什麼這個問題被分配給我，以便判斷自己是否真的適合回覆。

## 6. 觀望者狀態的處理方式

| 使用狀態 | 放入位置 | 設計含意 |
|---|---|---|
| 先確認安全再互動 | P-01, P-02 | 需要清楚匿名、隱私、安全審核、檢舉/封鎖說明。 |
| 先理解信箱和論壇差異 | P-01 | 需要導覽或範例說明一對一信箱不是公開發文等留言。 |
| 先判斷自己是否適合回覆 | P-02 | 需要問題摘要、配對理由、拒絕/略過機制。 |
| 先觀察是否有人回覆 | P-01 | 需要等待狀態、替代內容或候補回覆機制。 |

結論：`觀望者` 不新增為第三 Persona，因為目前資料更支持它是一種進入平台前後的使用狀態，而不是穩定角色類型。

## 7. 兩類 Persona 的共同設計原則

| 原則 | 對應洞察 | 說明 |
|---|---|---|
| 匿名是預設條件 | IN-01 | 兩類 Persona 都需要安全匿名，而不是把匿名當作單獨族群。 |
| 先處理安全與信任 | IN-01 | 詐騙、個資、惡意互動、錯誤建議會直接影響參與意願。 |
| 以相關經驗連結雙方 | IN-02 | 尋求者想找有相關經驗的人，分享者也需要判斷問題是否適合自己回覆。 |
| 優先非即時互動 | IN-03 | 非即時信箱能降低壓力，也方便加入平台提醒與審核。 |
| 流程要簡單清楚 | IN-04, IN-05 | 導覽與簡化入口可協助新使用者理解如何提問、回覆與保持安全。 |
| 信箱假設需被驗證 | IN-02, IN-03, IN-04 | 一對一信箱需要驗證差異化、等待、配對品質與分享者回覆誘因。 |

## 8. 一對一匿名經驗交流信箱下的 Persona 風險

| 風險 | 主要影響 Persona | 說明 | 後續驗證方向 |
|---|---|---|---|
| 看不出和論壇/社群差異 | P-01, P-02 | 使用者可能覺得匿名發文、配對、AI 等既有平台也有。 | 概念測試中比較信箱、公開論壇、經驗瀏覽三種敘事。 |
| 沒人回或等待太久 | P-01 | 尋求者投遞後若沒有回覆，信任會下降。 | 測試等待狀態、候補回覆、導覽內容補位。 |
| 配對不對稱 | P-01, P-02 | 提問者想問的主題可能和分享者能回答的經驗不匹配。 | 測試主題、人生經驗、回覆風格的配對欄位。 |
| 分享者誘因不足 | P-02 | 分享者可能沒有理由持續回覆陌生人。 | 測試成就感、感謝回饋、低負擔回覆流程。 |
| 回覆責任壓力 | P-02 | 分享者擔心建議被誤用或被過度依賴。 | 加入非專業建議提示與回覆界線。 |

## 9. 進入 04 User Stories 的前提

進入 `04_User Stories` 時，應以本文件的兩類 Persona 作為主要角色來源，並維持以下限制：

- 每則 User Story 需能回到 Persona 與 `F/T/IN` 證據來源。
- User Stories 可使用本文件列出的故事線索，但仍需另行收斂為 6-8 則核心故事。
- 一對一匿名經驗交流信箱仍是主要產品假設，後續故事與 MVP 排序需保留差異化、等待、配對品質與分享者誘因風險。
- 安全、隱私、檢舉/封鎖、回覆界線與非專業建議提示應作為匿名交流的基礎條件，不宜被視為可完全延後的附加功能。
