# 調整後問卷重點整理產出計畫

## Summary

本計畫用於產出 `01_問卷重點整理.md`。文件定位是開發前的可能使用者分析，不是完整 UX 研究報告。問卷樣本數為 55，未加入訪談，因此所有結論都應寫成探索性需求線索。

原先計畫把「一對一匿名經驗交流信箱」視為主要方向，但匿名提問、匿名回覆與匿名信箱本質上都屬於匿名交流。因此調整後的整理方式是：

- 先確認「匿名交流」是否是核心需求方向。
- 再比較匿名交流的不同互動形式。
- 不在問卷重點整理階段提前決定唯一 MVP 主流程。
- 將一對一信箱定位為匿名交流的候選形式之一，後續再由 User Stories 與 MVP 排序決定優先級。

## Key Adjustments

- 核心結論改為「匿名交流需求明顯」，而不是「一對一信箱必須作為唯一主流程」。
- 匿名文字信箱、公開匿名提問、一對一匿名信箱、經驗文章/故事瀏覽都列為候選互動形式。
- 問卷重點整理只呈現各形式的支持度、風險與適用情境。
- 後續 User Stories 可同時保留匿名提問、匿名信箱、經驗瀏覽三組候選故事。
- 安全與信任機制需跨所有匿名交流形式呈現。

## File Structure

`01_問卷重點整理.md` 固定為 6 段：

1. `樣本與使用限制`
   - 樣本數、年齡分布、資料限制。
   - 明確說明此問卷只作為開發前需求線索。

2. `核心量化發現`
   - 用 `F-xx` 編號整理重要數據。
   - 每個發現都包含資料結果、初步解讀、後續可支撐的方向。

3. `匿名交流形式比較`
   - 比較經驗文章/故事、匿名文字信箱、公開匿名提問、一對一配對聊天等形式。
   - 不提前把某一種形式定為唯一主流程。

4. `需求與使用誘因`
   - 整理使用者想請教的問題類型、重視條件、提高使用意願的因素。
   - 聚焦後續可轉成 User Stories 的需求。

5. `安全與信任風險`
   - 整理詐騙、個資、被說教、外流、內容可信度等風險。
   - 說明安全機制是匿名交流的核心條件。

6. `開放題主題與後續故事線索`
   - 用 `T-xx` 編號整理開放題主題。
   - 每個主題附簡短代表摘要與可支撐的後續 User Story 類型。

## Trace Format

後續 User Stories 可引用兩種 ID：

- `F-xx`：量化發現，例如匿名提問意願、匿名回覆意願、安全審核接受度。
- `T-xx`：開放題主題，例如詐騙、個資外洩、內容品質、平台規範。

建議 User Story 證據卡至少引用：

- 1 個量化發現 `F-xx`。
- 1 個風險或開放題主題 `T-xx`，若該故事涉及安全、信任或內容品質。

## Human Review Points

產出後需要人為確認：

- 是否同意把「匿名交流」作為核心方向，而不是只把「一對一信箱」作為核心方向。
- 是否接受在問卷重點整理中只比較互動形式，不先決定 MVP 優先級。
- 是否同意後續 User Stories 同時保留匿名提問、匿名信箱、經驗瀏覽三組候選故事。
- 是否要排除明顯離題或玩笑性回答，避免影響主題判讀。
- 是否維持探索性語氣，例如「目前資料顯示」「初步推測」「可作為後續設計假設」。

## Test Plan

- 檢查每個 `F-xx` 是否有明確數字或比例。
- 檢查每個 `T-xx` 是否來自開放題資料。
- 檢查文件是否避免提前把一對一信箱寫成唯一主流程。
- 檢查後續 User Stories 是否能引用 `F/T` 編號。
- 檢查安全與信任風險是否跨所有匿名交流形式呈現。
- 檢查文件語氣是否適合非 UI/UX 背景團隊向老師或上級解釋。

## Assumptions

- 產出檔名為 `01_問卷重點整理.md`。
- 文件以 Markdown 交付，之後可貼到 Notion。
- 問卷資料來源為 `跨世代／跨人生階段交流平台需求調查v2 (回覆).xlsx`。
- 這份文件只整理問卷重點，不產出 Persona、User Stories 或 MVP 排序。
