> **Google Sheets 做資料與證據**
> **FigJam 做協作與收斂**
> **Notion 做報告與最終整理**

**Sheets 是資料來源，FigJam 是分析工作坊，Notion 是最後可交付的研究文件。**

- Google Sheets 適合多人即時協作與權限控管，Google 官方說明可設定 Editor、Commenter、Viewer 等權限，也支援多人共同編輯試算表。([Google Help][1])
- FigJam 是 Figma 官方提供的線上協作白板，適合團隊 diagram、brainstorm、整理想法與設計協作。([Figma][2])
- Notion 則適合把研究內容整理成結構化頁面，且可透過分享與權限設定控制團隊成員可編輯或檢視的內容。([Notion][3])

# 一、三個工具的角色定位

## 1. Google Sheets：資料與證據中心

**定位：唯一可信資料來源。**

Google Sheets 不只是放問卷原始資料，而是用來建立「所有研究結論的證據鏈」。也就是說，Persona、User Story、MVP 功能不能只從 FigJam 或 Notion 想出來，都應該能回到 Sheets 找到依據。

### 建議用途

| 工作     | Google Sheets 中的用途            |
| ------ | ----------------------------- |
| 問卷統計   | 整理每題比例、次數、摘要                  |
| 交叉分析   | 建立 3 組重點交叉分析                  |
| 開放題編碼  | 為每則開放題標記 code / theme         |
| 證據追溯   | 將 Persona、洞察、User Story 對應回資料 |
| MVP 依據 | 功能排序時記錄支持資料與信心等級              |

### 不建議用途

| 不建議在 Sheets 做 | 原因         |
| ------------- | ---------- |
| Persona 視覺化   | 不適合閱讀與團隊討論 |
| 親和圖分群         | 試算表不如白板直覺  |
| 最終報告          | 可讀性與敘事性較弱  |

---

## 2. FigJam：協作與收斂中心

**定位：團隊一起把資料變成洞察的地方。**

FigJam 應該用於「討論、分群、收斂」，而不是當作正式報告。它最適合親和圖、洞察句、Persona 草稿、JTBD、User Stories、MVP 排序。

FigJam 官方也提供 affinity diagram template，可用於把想法、意見與問題依自然關係分組；這正好對應你們 Day 1 的親和圖工作。([Figma][4])

### 建議用途

| 工作           | FigJam 中的用途                 |
| ------------ | --------------------------- |
| 親和圖          | 將問卷發現與開放題代碼貼成便利貼            |
| 洞察句          | 每個群組轉成「使用者需要……因為……」         |
| Persona 草稿   | 團隊共編 2–3 張 Persona          |
| JTBD         | 將 Persona 轉成使用者任務           |
| User Stories | 共同拆解功能需求                    |
| MVP 排序       | 使用 MoSCoW 或 Impact × Effort |

### 不建議用途

| 不建議在 FigJam 做 | 原因           |
| ------------- | ------------ |
| 原始問卷資料        | 容易失去資料完整性    |
| 大量統計表         | 白板不適合計算      |
| 最終正式報告        | 內容會較分散，不適合提交 |

---

## 3. Notion：報告與交付中心

**定位：整理最終研究成果與決策紀錄。**

Notion 應該作為最後交付用的研究報告。它可以整理方法、限制、主要發現、Persona、User Stories、MVP 功能排序與待驗證假設。

Notion 的資料庫頁面可設定「Can edit content」權限，讓成員能新增、編輯資料庫內容，但不能修改資料庫屬性或視圖，這對控制報告結構很有幫助。([Notion][5])

### 建議用途

| 工作           | Notion 中的用途              |
| ------------ | ------------------------ |
| 研究方法說明       | 說明為何採探索性流程               |
| 研究限制         | 樣本數、未訪談、非正式分群            |
| 主要發現         | 整理 3–5 個研究洞察             |
| Persona      | 放入定稿版 Persona            |
| User Stories | 用資料庫管理故事、優先級與驗收條件        |
| MVP 清單       | 記錄 Must / Should / Could |
| 待驗證假設        | 作為後續研究 backlog           |

### 不建議用途

| 不建議在 Notion 做 | 原因               |
| ------------- | ---------------- |
| 問卷原始分析        | 計算與交叉分析不如 Sheets |
| 親和圖           | 白板互動不如 FigJam    |
| 大量多人同步拖拉整理    | FigJam 更適合       |

# 四、FigJam 建議白板結構

建議 FigJam 不要開多個檔案，只開一個：

**《2 天研究衝刺工作坊》**

---

## 區塊 1：Sprint Goal & Roles

```text
研究目標：
將已清理完成的 50–60 份問卷，轉換成假設型 Persona、JTBD、User Stories 與 MVP 功能排序。

A：研究整合與品質把關
B：資料統計與證據追溯
C：開放題、競品與產品轉換
```

---

## 區塊 2：Key Survey Findings

由 B 從 Sheets 搬過來，不要貼太多，只放重點。

```text
F-01：匿名交流相關需求明顯
F-02：使用者對隱私與安全有顧慮
F-03：部分使用者想聽過來人經驗
F-04：跨世代交流有吸引力但可能有尷尬／被說教風險
F-05：操作簡單與清楚引導可能影響使用意願
```

每張便利貼建議格式：

```text
[F-01 / Sheets Summary]
觀察內容：
可能設計意義：
```

---

## 區塊 3：Open Coding Themes

由 C 從 Sheets 搬主題，不搬全部原文。

```text
T-01 匿名與曝光顧慮
T-02 低壓力互動
T-03 經驗參考需求
T-04 跨世代溝通顧慮
T-05 信任與安全
T-06 分享者界線
T-07 操作負擔
```

---

## 區塊 4：Affinity Map

三人一起做。

### 便利貼格式

```text
[來源 ID] 觀察
```

範例：

```text
[F-01] 匿名交流需求明顯
[T-01] 怕被熟人知道
[T-05] 擔心個資外洩
[競品-01] 匿名平台通常提供檢舉或封鎖
```

### 分群目標

整理成 5–7 群：

| 群組    | 可能洞察           |
| ----- | -------------- |
| 匿名安全感 | 低曝光表達空間        |
| 經驗參考  | 過來人經驗比單純聊天更有價值 |
| 低壓力互動 | 需要避免批判、說教、尷尬   |
| 信任與風險 | 需要安全、檢舉、封鎖     |
| 操作負擔  | 需要清楚簡單流程       |
| 分享者界線 | 分享者需要保護與回覆規範   |

---

## 區塊 5：Insight Statements

每群最後要寫成洞察句。

### 模板

```text
使用者需要【需求】，
因為【痛點／情境】，
因此產品應該【設計方向】。
證據來源：【Sheets ID / Theme ID / 競品 ID】
信心等級：【高／中／低】
```

### 範例

```text
使用者需要低曝光的表達空間，
因為現實人際關係會讓他們擔心被認出或被評價，
因此產品應提供匿名發問、暱稱與清楚的隱私說明。
證據來源：F-01、T-01、T-05
信心等級：中高
```

---

| 經驗尋求者   | 必做             |

# 十二、建議你們直接採用的工作原則

1. **Sheets 不放結論，只放資料與證據。**
2. **FigJam 不放原始資料，只放可討論的觀察與洞察。**
3. **Notion 不做即時腦暴，只放定稿與決策紀錄。**
4. **每個 Persona 都要有證據來源 ID。**
5. **每個 User Story 都要對應 Persona 與洞察 ID。**
6. **每個 Must-have 功能都要有使用者價值與風險標註。**

這樣分工後，你們 3 人在 2 天內的工作會比較不混亂，也能讓最後報告看起來有研究依據，而不是單純團隊討論結果。
