# 03_假設型 Persona：程式產出計畫

## Summary

本文件規劃如何依 `02_需求洞察整理.md` 產出 2 類假設型 Persona 草稿。此階段只處理 `Progress-decision.md` 中「假設型 Persona 可由程式產出」的部分：

- 根據已決定的方向產生 Persona 草稿。
- 套用固定欄位。
- 將問卷數據、開放題主題與需求洞察填入證據欄。

Persona 仍需團隊人工定稿。

## Persona 原則

- 只做 2 類 Persona：`經驗尋求者`、`經驗分享者`。
- 匿名不是 Persona 類型，而是平台預設交流條件。
- Persona 不寫成真實市場族群，只寫成開發前假設型角色。
- 每張 Persona 都必須追溯到 `F-xx`、`T-xx` 與 `IN-xx`。

## 欄位模板

| 欄位 | 說明 |
|---|---|
| Persona ID | P-01 / P-02 |
| Persona 名稱 | 經驗尋求者 / 經驗分享者 |
| 一句話描述 | 用非 UX 術語說明角色 |
| 代表情境 | 什麼情況下會使用平台 |
| 核心需求 | 對應正式洞察 |
| 主要痛點 | 來自問卷與開放題 |
| 可能行為 | 提問、回覆、導覽瀏覽、觀望等 |
| 產品機會 | 可支撐的功能方向 |
| 證據來源 | `F-xx`, `T-xx`, `IN-xx` |
| 待驗證假設 | 後續訪談或測試要確認的問題 |

## 程式可產出內容

### P-01 經驗尋求者

- 面對職涯、人生方向、人際或情緒議題時，希望找到有相關經驗的人提供參考。
- 共同需求：安全匿名、相關經驗、非即時互動、清楚流程。
- 對應洞察：`IN-01`, `IN-02`, `IN-03`, `IN-04`, `IN-05`。
- 可支撐故事：匿名提問、主題選擇、非即時匿名信箱、導覽式輔助入口、安全提醒。

### P-02 經驗分享者

- 願意匿名分享自身經驗，但需要清楚互動邊界、安全保護與回覆規範。
- 共同需求：匿名保護、回覆規範、避免被誤解或承擔過度責任。
- 對應洞察：`IN-01`, `IN-02`, `IN-03`, `IN-05`。
- 可支撐故事：匿名回覆、回覆前提醒、希望回覆方式、檢舉/封鎖、平台內互動限制。

## Human Review Points

- 確認 `經驗尋求者` 與 `經驗分享者` 的角色邊界是否清楚。
- 確認 Persona 是否太像功能清單，需要補更多使用情境。
- 確認是否需要把「觀望者」作為狀態，而不是第三類 Persona。
- 確認每張 Persona 是否都能導出至少 2 則 User Stories。
- 確認語氣是否保持假設型，不寫成已驗證使用者。

## Test Plan

- Persona 不得把「匿名」寫成一類使用者。
- 每張 Persona 至少對應 2 個正式洞察。
- 每張 Persona 至少引用 2 個以上 `F/T` 證據來源。
- 每張 Persona 都要有待驗證假設。
- 每張 Persona 都要能支撐後續 User Stories。

## Assumptions

- `02_需求洞察整理.md` 是 Persona 的主要依據。
- 本階段不產出正式 User Stories，只標示可支撐方向。
- Persona 定稿仍需人工確認。
