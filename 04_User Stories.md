# 04_User Stories

## 1. 文件定位與使用限制

本文件由 `04_User Stories初稿.md` 的人工審核結果整理而成，作為進入 `05_MVP 建議與待驗證假設` 前的 User Stories 版本。`04_User Stories初稿.md` 保留為初稿與人工審核紀錄。

本文件仍屬開發前的假設型使用者分析，不是最終開發規格，也不是已驗證需求。問卷尚未搭配訪談或可用性測試，因此後續 MVP 排序仍需保留待驗證假設。

使用限制：

- 一對一匿名經驗交流信箱是團隊目前選定的主要產品假設，不是問卷已直接證明的唯一最佳形式。
- 本文件整理 7 則人工審核後核心 User Stories，符合 6-8 則核心故事門檻。
- `US-04 查看等待狀態與替代建議` 已由人工審核暫定延後到 05 MVP，不列入本階段核心故事，但保留為後續待驗證假設。
- 每則核心故事需能回到 Persona 與 `F/T/IN` 證據來源。

## 4. 核心 User Story 證據卡

### US-01 新手導覽與信箱差異理解

| 欄位 | 內容 |
|---|---|
| Persona | P-01 經驗尋求者 / P-02 經驗分享者 |
| User Flow | 瀏覽 / 導覽 |
| User Story | 作為一位第一次使用平台的人，我希望可以看到簡短導覽與範例，以便理解一對一匿名經驗信箱和一般論壇發文有什麼不同，並知道自己可以作為提問者或分享者參與。 |
| 支持證據 | F-06 看別人的經驗文章或故事 35 人，53.8%；F-12 先瀏覽內容 24 人，36.9%；F-13 明確下一步提示 40 人，61.5%；T-06 冷啟動/沒人回；T-08 平台差異化/配對與回覆誘因；IN-04, IN-05。 |
| 外部方法對應 | 符合 User Story 需描述使用者、目標與價值；驗收條件補足可判斷的成功狀態。 |
| 初步驗收條件 | 使用者第一次進入時能看到平台用途、匿名與安全原則、一對一信箱和論壇差異；導覽後能選擇體驗一次提問者或分享者流程，以快速理解平台功能。 |
| 風險與待驗證假設 | 導覽是否真的降低第一次使用門檻仍待驗證；若導覽太長，可能讓使用者更不想開始。 |
| 人工審核結果 | 保留，並更新 User Story 及初步驗收條件。 |

### US-02 匿名投遞一對一經驗信箱

| 欄位 | 內容 |
|---|---|
| Persona | P-01 經驗尋求者 |
| User Flow | 提問 |
| User Story | 作為一位經驗尋求者，我希望可以匿名投遞一對一經驗信箱，以便在不暴露身份的情況下尋求有相關經驗者的參考。 |
| 支持證據 | F-01 匿名提問 4 分以上 45 人，69.2%；F-03 一對一匿名經驗交流信箱 4 分以上 36 人，55.4%；F-12 匿名提問 39 人，60.0%；F-14 擔心詐騙/騷擾/惡意言論 42 人，64.6%；F-15 擔心個資外洩 40 人，61.5%；T-02 個資/隱私/外流；IN-01, IN-03。 |
| 外部方法對應 | 以使用者目標與理由描述故事，再用驗收條件補足匿名與安全的成功條件。 |
| 初步驗收條件 | 使用者能在平台內匿名建立問題；送出前能看到匿名與隱私提醒；問題不要求填入真實姓名或私人聯絡方式；送出後能看到後續狀態或下一步。 |
| 風險與待驗證假設 | 一對一信箱不是問卷中支持度最高的互動形式，仍需驗證是否比論壇、匿名社群或內容瀏覽更有價值。 |
| 人工審核結果 | 保留，目前為此平台核心功能。 |

### US-03 選擇主題與希望回覆方式

| 欄位 | 內容 |
|---|---|
| Persona | P-01 經驗尋求者 |
| User Flow | 提問 / 媒合 |
| User Story | 作為一位經驗尋求者，我希望可以選擇問題主題與希望回覆方式，以便提高回覆和自己情境的相關性。 |
| 支持證據 | F-10 職涯選擇 34 人，52.3%，人生方向 23 人，35.4%，人際關係 20 人，30.8%；F-11 年齡不重要、相關經驗更重要 41 人，63.1%；F-12 使用者重視安全、匿名、非即時與先瀏覽；T-08 平台差異化/配對與回覆誘因；IN-02, IN-03。 |
| 外部方法對應 | 故事聚焦使用者希望達成的結果，避免在 User Story 階段把媒合演算法或後台規則寫成定案。 |
| 初步驗收條件 | 使用者能選擇至少一個問題主題；能選擇希望收到的回覆風格，例如分享類似經驗或提供不同觀點；送出前能確認選擇內容。 |
| 風險與待驗證假設 | 使用者認為「相關經驗」需要如何描述仍待驗證；細緻經驗條件、後台演算法配對與配對權重需延後到 05 MVP 排序與後續設計判斷。 |
| 人工審核結果 | 保留；第一版只先保留主題與回覆方式，細緻經驗條件可由後台演算法配對或延後到 05 MVP 排序。 |

### US-05 分享者收到相關提問與配對理由

| 欄位 | 內容 |
|---|---|
| Persona | P-02 經驗分享者 |
| User Flow | 媒合 / 回覆前判斷 |
| User Story | 作為一位經驗分享者，我希望可以收到與我經驗相關的一對一匿名提問，並知道為什麼這個問題被分配給我，以便判斷自己是否真的適合回覆。 |
| 支持證據 | F-02 匿名回覆他人煩惱 4 分以上 54 人，83.1%；F-11 相關經驗比年齡更重要 41 人，63.1%；F-17 不知道對方背景是否可信 20 人，30.8%；T-03 內容品質/錯誤建議/不實用；T-08 平台差異化/配對與回覆誘因；IN-02, IN-03。 |
| 外部方法對應 | 故事描述分享者的判斷目標，不把媒合技術細節提前定案。 |
| 初步驗收條件 | 分享者能看到問題摘要、主題與必要背景；能看到簡短配對理由；能在回覆前判斷是否符合自身經驗，再選擇是否回覆；不得揭露提問者可識別個資。 |
| 風險與待驗證假設 | 配對理由是否足以建立信任仍待驗證；若理由太模糊，分享者可能不願回覆。 |
| 人工審核結果 | 保留，並更新初步驗收條件。 |

### US-06 分享者匿名回覆並保留拒絕或略過權

| 欄位 | 內容 |
|---|---|
| Persona | P-02 經驗分享者 |
| User Flow | 回覆 |
| User Story | 作為一位經驗分享者，我希望可以匿名回覆，也可以略過、拒絕或結束不適合的交流，以便維持自己的參與界線。 |
| 支持證據 | F-02 匿名回覆意願高；F-12 安全審核與檢舉 40 人，61.5%；F-14 擔心詐騙/騷擾/惡意言論 42 人，64.6%；T-01 詐騙/惡意/廣告/機器人；T-04 平台規範/安全機制；T-08 平台差異化/配對與回覆誘因；IN-01, IN-03。 |
| 外部方法對應 | User Story 保留使用者價值，acceptance criteria 補充可判斷的互動界線。 |
| 初步驗收條件 | 分享者收到提問清單後，可先查看問題摘要、主題、必要背景與簡短配對理由，再選擇要匿名回覆哪一個提問；略過提問不會被設計成懲罰，且提問者不會知道自己被特定分享者拒絕回覆；回覆後能選擇結束互動，或根據提問者反饋決定是否繼續後續交流。 |
| 風險與待驗證假設 | 分享者是否願意持續回覆仍待驗證；雖然尋求者不會知道自己被略過，但被過多人略過可能造成尋求者等待過久。 |
| 人工審核結果 | 保留，並調整初步驗收條件及風險與待驗證假設。 |

### US-07 回覆前友善提醒與非專業建議界線

| 欄位 | 內容 |
|---|---|
| Persona | P-02 經驗分享者 |
| User Flow | 安全 / 回覆品質 |
| User Story | 作為一位經驗分享者，我希望回覆前能看到友善提醒與非專業建議界線，以便避免說教、錯誤建議或讓對方過度依賴我的回覆。 |
| 支持證據 | F-16 被對方評價或說教 27 人，41.5%；F-19 對方提供錯誤或不適合的建議 16 人，24.6%；T-03 內容品質/錯誤建議/不實用；T-05 被批判/說教/情感內耗；IN-01, IN-03。 |
| 外部方法對應 | 驗收條件描述成功狀態，避免直接指定完整審核或 AI 實作。 |
| 初步驗收條件 | 分享者回覆前能看到簡短回覆規範；提醒以自身經驗分享為主，不提供專業診斷或保證；提醒避免說教、攻擊或要求交換私人聯絡方式；使用者能在不離開流程的情況下繼續回覆。 |
| 風險與待驗證假設 | 提醒是否能有效降低說教與錯誤建議仍待驗證；提醒太頻繁可能造成回覆負擔。 |
| 人工審核結果 | 保留。 |

### US-08 檢舉、封鎖或結束不舒服互動

| 欄位 | 內容 |
|---|---|
| Persona | P-01 經驗尋求者 / P-02 經驗分享者 |
| User Flow | 安全 / 互動結束 |
| User Story | 作為一位平台使用者，我希望可以檢舉、封鎖或結束讓我不舒服的互動，以便降低匿名交流中的詐騙、騷擾與隱私風險。 |
| 支持證據 | F-04 AI 協助檢查不當內容 4 分以上 48 人，73.8%；F-12 安全審核與檢舉 40 人，61.5%；F-14 詐騙/騷擾/惡意言論 42 人，64.6%；F-15 個資外洩 40 人，61.5%；T-01 詐騙/惡意/廣告/機器人；T-02 個資/隱私/外流；T-04 平台規範/安全機制；IN-01。 |
| 外部方法對應 | 故事保留使用者安全價值，驗收條件用於確認安全能力是否可被使用者找到與操作。 |
| 初步驗收條件 | 使用者能在互動中找到檢舉、封鎖或結束交流入口；檢舉時可選擇原因；結束交流後不需繼續回覆對方；平台需提示不要交換私人聯絡方式或敏感資訊。 |
| 風險與待驗證假設 | 哪些安全機制最能降低疑慮仍待驗證；檢舉與封鎖流程若太複雜，使用者可能不會使用。 |
| 人工審核結果 | 保留。 |
