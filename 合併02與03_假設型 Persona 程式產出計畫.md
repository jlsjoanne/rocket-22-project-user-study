# 合併 02_需求洞察整理與規劃 03_假設型 Persona 程式產出

## Summary

先將 `02_需求洞察整理初稿.md` 與 `02_需求洞察整理人工收斂結果.md` 合併為正式版 `02_需求洞察整理.md`，以人工收斂結果為準。正式洞察保留 5 個，並標註資料仍屬探索性線索。

接著依 `Progress-decision.md` 的「3. 假設型 Persona」可由程式產出範圍，規劃下一階段 Persona 草稿。Persona 固定為 2 類：`經驗尋求者`、`經驗分享者`；匿名不作為 Persona，匿名是平台預設交流條件。

## 02 合併方式

`02_需求洞察整理.md` 內容結構：

- `資料來源與限制`
  - 引用 `01_問卷重點整理.md`、問卷資料、初稿與人工收斂結果。
  - 說明此文件是「程式初稿 + 人工收斂」後的正式需求洞察整理。

- `正式需求洞察總覽`
  - `IN-01 安全可信的交流環境`
    - 由 `IN-DRAFT-01` 與 `IN-DRAFT-06` 合併。
    - 作為後續設計必須優先滿足的核心需求。
  - `IN-02 以相關經驗為核心的交流對象`
  - `IN-03 低壓力、非即時的互動方式`
  - `IN-04 導覽式輔助入口`
    - 由原「先瀏覽內容再決定是否互動」調整而來。
    - MVP 中定位為第一次使用時的導覽/範例入口，不排除後續發展為經驗瀏覽功能。
  - `IN-05 清楚簡單的新手流程`

- `每個洞察的證據卡`
  - 欄位固定為：證據來源、資料看到什麼、需求假設、可支撐 User Stories、產品設計含意、待驗證問題。
  - 使用 `F-xx` 與 `T-xx` 編號，方便後續 User Story 證據卡引用。

- `人工收斂決策紀錄`
  - 記錄已確認事項：
    - 匿名交流是共同方向。
    - 產品重點調整為「跨人生階段／相關經驗」。
    - 安全可信是後續設計必須優先滿足的核心需求。
    - 經驗瀏覽在 MVP 中先作為導覽式輔助入口。

## 03 Persona 程式產出規劃

下一階段建議產出兩份文件：

- `03_假設型 Persona：程式產出計畫.md`
  - 保存 Persona 產出方法、欄位、資料來源與限制。
- `03_假設型 Persona.md`
  - 產出 2 類 Persona 草稿，供團隊人工定稿。

Persona 欄位固定：

| 欄位 | 說明 |
|---|---|
| Persona ID | P-01 / P-02 |
| Persona 名稱 | 經驗尋求者 / 經驗分享者 |
| 一句話描述 | 用非 UX 術語說明角色 |
| 代表情境 | 什麼情況下會使用平台 |
| 核心需求 | 對應 `IN-xx` 洞察 |
| 主要痛點 | 來自 `F/T` 證據 |
| 可能行為 | 提問、回覆、瀏覽、觀望等 |
| 產品機會 | 可支撐的功能方向 |
| 證據來源 | `F-xx`, `T-xx`, `IN-xx` |
| 待驗證假設 | 後續訪談或測試要確認的問題 |

程式可產出的 Persona 草稿：

- `P-01 經驗尋求者`
  - 面對職涯、人生方向、人際或情緒議題時，希望找到有相關經驗的人提供參考。
  - 共同需求：安全匿名、相關經驗、非即時互動、清楚流程。
  - 對應洞察：`IN-01`, `IN-02`, `IN-03`, `IN-04`, `IN-05`。

- `P-02 經驗分享者`
  - 願意匿名分享自身經驗，但需要清楚互動邊界、安全保護與回覆規範。
  - 共同需求：匿名保護、回覆規範、避免被誤解或承擔過度責任。
  - 對應洞察：`IN-01`, `IN-02`, `IN-03`, `IN-05`。

## Test Plan

- `02_需求洞察整理.md` 必須保留 5 個正式洞察，不再保留 6 個 `IN-DRAFT` 作為正式結果。
- `IN-01` 必須明確寫成「後續設計必須優先滿足的核心需求」。
- `IN-04` 必須寫成「MVP 導覽式輔助入口」，不是完整內容瀏覽功能。
- 每個正式洞察至少引用 2 個以上 `F/T` 證據來源。
- Persona 不得把「匿名」寫成一類使用者。
- 每個 Persona 至少對應 2 個正式洞察與 2 個以上問卷證據來源。

## Assumptions

- `02_需求洞察整理人工收斂結果.md` 的人工決策優先於初稿。
- `02_需求洞察整理.md` 是正式合併版；初稿與人工收斂結果保留作為歷史紀錄。
- 下一階段 Persona 仍是程式產出草稿，正式 Persona 需要人工確認。
