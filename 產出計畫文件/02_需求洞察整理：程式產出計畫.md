# 02_需求洞察整理：程式可產出部分計畫

## Summary

根據 `Progress-decision.md`，本階段只產出「需求洞察整理」中可由程式輔助完成的部分：開放題初步分類、主題次數、量化發現與主題對照、初版需求主題。

不在此階段定稿洞察，也不決定最終 MVP 優先級。

建議後續輸出檔名：`02_需求洞察整理.md`。

## File Structure

`02_需求洞察整理.md` 建議包含：

- `資料來源與限制`：引用 `01_問卷重點整理.md`，說明此文件是程式輔助初稿。
- `初步需求主題總覽`：列出 6 個 `IN-DRAFT`。
- `每個主題的證據卡`：包含資料看到什麼、初步需求假設、可支撐 User Stories、需人為判讀。
- `主題關係整理`：標示哪些主題可合併，例如安全信任與回覆規範可能同屬平台治理。
- `下一步人工收斂建議`：由團隊決定最後保留 4-6 個正式洞察。

## Human Review Points

- 是否保留 6 個主題，或合併成 4-5 個。
- 是否將「安全可信的匿名交流」列為最高優先洞察。
- 是否把「相關經驗」放在「跨世代」之前作為產品核心。
- 是否把「非即時信箱」視為主要互動候選，而不是即時配對聊天。
- 是否把「經驗瀏覽」作為降低冷啟動風險的輔助入口。

## Test Plan

- 每個 `IN-DRAFT` 至少引用 2 個以上 `F/T` 證據來源。
- 每個主題都能對應至少 1 種後續 User Story。
- 文件需明確標註「初版需求主題，尚待人工收斂」。
- 避免把程式分類結果寫成已驗證洞察。
- 保持工程師可解釋的語氣：資料看到什麼、可能代表什麼、可支撐什麼功能。

## Assumptions

- 來源文件使用 `01_問卷重點整理.md`。
- 不重新決定 MVP 優先級。
- 不在此步產出 Persona 或 User Stories。
