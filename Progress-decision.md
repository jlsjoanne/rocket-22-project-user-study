# 流程自動化與人為判讀分工

可以把流程分成三類：**程式可直接產出**、**程式輔助但需人判讀**、**必須人為決策**。

## 一、總覽

| 產出文件 | 程式可產出 | 需人為介入 |
|---|---|---|
| `01_問卷重點整理` | 高 | 低 |
| `02_需求洞察整理` | 中 | 高 |
| `03_假設型 Persona` | 中 | 高 |
| `04_User Stories` | 中 | 高 |
| `05_MVP 建議與待驗證假設` | 中 | 高 |

## 二、各文件分工判斷

### 1. 問卷重點整理

可由程式產出：

- 讀取 Excel 問卷資料。
- 計算樣本數、年齡分布、身分分布。
- 統計單選題比例。
- 統計多選題被選次數與比例。
- 計算量表題平均值、中位數、4 分以上比例。
- 擷取開放題回答清單。
- 產出 Google Sheets / Markdown 表格格式。

需要人為判讀：

- 判斷哪些數據「重要」。
- 避免把小樣本結果寫成市場定論。
- 判斷異常回答是否保留，例如離題或玩笑回答。

結論：這份文件最適合自動化，人工只做最後檢查。

### 2. 需求洞察整理

可由程式產出：

- 將開放題依關鍵字初步分類，例如詐騙、個資、被說教、內容品質、冷啟動。
- 統計每個主題出現次數。
- 將相關數據與開放題主題放在一起。
- 產出初版需求主題，例如安全信任、低壓力互動、經驗參考、非即時交流。

需要人為判讀：

- 決定最後保留哪 4-6 個需求主題。
- 判斷不同主題是否該合併。
- 將「使用者說了什麼」轉成「可能需求是什麼」。
- 避免過度解讀開放題。

結論：程式可以做初步分群與摘要，但洞察命名與收斂必須由人判斷。

### 3. 假設型 Persona

可由程式產出：

- 根據已決定的方向產生 Persona 草稿。
- 套用固定欄位：名稱、情境、需求、痛點、行為、產品機會、證據來源、待驗證假設。
- 將問卷數據與開放題主題填入 Persona 證據欄。

需要人為判讀：

- 確認 Persona 是否真的符合團隊產品方向。
- 決定「經驗尋求者」與「經驗分享者」的邊界。
- 避免把匿名寫成 Persona，因為已決定匿名是平台 default。
- 判斷 Persona 是否太像功能清單，而不是使用者角色。

結論：程式可產出草稿，但 Persona 定稿需要團隊討論。

### 4. User Stories

可由程式產出：

- 依 Persona 與核心流程產出 6-8 則 User Story 草稿。
- 套用格式：作為一位……我希望……以便……
- 建立證據卡欄位。
- 初步補上支持數據、代表回答、驗收條件、風險。
- 將故事對應到 user flow，例如提問、媒合、回覆、安全、瀏覽。

需要人為判讀：

- 決定哪些故事算核心。
- 判斷故事是否真的支撐一對一匿名信箱。
- 調整故事粒度，避免太大或太細。
- 確認驗收條件是否合理。
- 排除證據不足或偏離 MVP 的故事。

結論：程式適合產出初版 backlog，人要負責挑選與修正。

### 5. MVP 建議與待驗證假設

可由程式產出：

- 根據問卷支持度與風險產生 Must / Should / Could 初稿。
- 標註每個功能的支持數據。
- 產出待驗證假設，例如匿名是否提升提問意願、一對一信箱是否有信任風險。
- 整理簡短競品觀察欄位。

需要人為判讀：

- 決定 Must-have 最終清單。
- 權衡使用者價值與開發成本。
- 判斷哪些功能第一版不做。
- 決定風險是否可接受。
- 對齊團隊開發能力與專題目標。

結論：程式可以提供排序建議，但 MVP 決策不能完全自動化。

## 三、依 Proposed Plan 流程拆解

| 流程 | 可程式產出 | 需人為介入 |
|---|---|---|
| 描述統計 | 幾乎可全自動 | 判斷重點數據 |
| 開放題快速整理 | 可做關鍵字分類與摘要 | 主題合併、命名、判讀 |
| 親和圖 | 可產出便利貼文字 | 分群與洞察收斂 |
| 2 類 Persona | 可產生草稿 | 定義角色邊界與定稿 |
| User Story 證據卡 | 可產生初稿 | 決定核心故事與驗收條件 |
| MVP 排序 | 可依數據與風險給建議 | 最終優先級決策 |
| 研究限制 | 可產生標準文字 | 確認語氣與報告定位 |
| User Flow / Wireframe 前置 | 可依故事列流程 | 畫面取捨與互動細節 |

## 四、建議自動化程度

最適合程式處理：

- 問卷統計。
- 開放題初步分類。
- 重要數據表。
- User Story 證據卡初稿。
- Persona 草稿。
- MVP 排序初稿。

最需要人處理：

- 需求洞察命名。
- Persona 定稿。
- 核心 User Stories 選擇。
- Must / Should / Could 最終排序。
- 風險判讀。
- 將結果轉成開發前共識。

簡單說：**程式負責整理、統計、產生草稿；人負責判讀、取捨、命名與決策。**
