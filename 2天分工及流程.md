# 2 天內最適合的流程

建議流程如下：

```text
已完成：問卷清理，確認無明顯無效問卷
↓
1. 描述統計與重點交叉分析
↓
2. 開放題快速主題分析
↓
3. 親和圖法整理痛點、需求與情境
↓
4. 桌面研究與競品／類似平台對照
↓
5. 建立洞察主題與需求假設
↓
6. 產出 2–3 個假設型 Persona
↓
7. 轉成 JTBD、User Stories 與 Acceptance Criteria
↓
8. MVP 功能排序與風險／待驗證標註
```

2 天內不建議產出 4 個以上 Persona。比較適合做 **2–3 個主要 Persona**，讓每個 Persona 都有足夠資料支撐，也比較容易轉成 User Stories 與 MVP。

---

# 3 人小組最適合分工

建議三人不要平均切成「你做 Persona、我做統計、他做簡報」這種方式，因為這樣容易造成分析斷裂。比較好的分工是：

| 角色               | 主要責任                                | 適合負責節點                     |
| ---------------- | ----------------------------------- | -------------------------- |
| A：研究整合與決策負責人     | 控制研究問題、整理洞察、確認不要過度推論                | 主題收斂、親和圖、Persona、風險標註      |
| B：問卷資料與量化整理負責人   | 處理選項題統計、交叉分析、圖表、數據摘要                | 描述統計、交叉分析、資料追溯表            |
| C：開放題／競品與產品轉換負責人 | 開放題編碼、競品對照、JTBD、User Stories、MVP 功能 | 主題分析、競品矩陣、User Stories、MVP |

但三人不能完全各做各的。最重要的節點，例如**親和圖、洞察假設、Persona 定稿、MVP 排序**，需要三人一起討論。因為 NN/g 將親和圖法視為能協作整理 UX 研究發現的方法，這一步正好可以讓團隊一起校準理解。([Nielsen Norman Group][2])

| 6:30–7:30 | 洞察句撰寫     | 將群組轉成洞察句           | 補上數據依據       | 補上代表語句        |

## Day 1 各節點執行重點與門檻

### 節點 1：描述統計

**執行重點**

B 負責整理所有核心題目的比例，包括：

| 題目類型 | 整理方式           |
| ---- | -------------- |
| 單選題  | 每個選項人數與比例      |
| 多選題  | 每個選項被選次數與比例    |
| 量表題  | 平均值、中位數、較高比例選項 |
| 背景題  | 年齡、身分、平台使用經驗分布 |

**A 要做的事**

A 要判斷哪些題目與 Persona 有關，例如匿名需求、隱私顧慮、跨世代交流意願、想獲得的幫助類型。

**注意事項**

不要把 50–60 份問卷寫成具市場代表性的結論。建議用語是：

> 本次探索性問卷中，填答者較明顯呈現出……

而不是：

> 大多數使用者都……

**進入下一節點門檻**

| 門檻                        |
| ------------------------- |
| 核心題目已整理成百分比或次數            |
| 已標出 5–8 個值得注意的數據點         |
| 已知道哪些題目與 Persona、MVP 功能有關 |
| 已避免使用「代表全部市場」的說法          |

---

### 節點 2：重點交叉分析

2 天內不建議做太多交叉分析。建議只做 3 組。

**最建議的 3 組**

| 交叉分析             | 用途                      |
| ---------------- | ----------------------- |
| 年齡／人生階段 × 匿名交流意願 | 判斷匿名需求是否集中在特定族群         |
| 煩惱類型 × 想獲得的幫助    | 判斷使用者是想要情緒支持、經驗分享還是實際建議 |
| 平台使用經驗 × 隱私／安全顧慮 | 判斷現有社群經驗是否影響信任需求        |

若你們問卷中有「是否願意分享經驗」的題目，可以把第三組改成：

| 交叉分析                   | 用途              |
| ---------------------- | --------------- |
| 願意分享經驗 × 對互動界線／安全機制的需求 | 判斷是否需要設計分享者保護機制 |

**注意事項**

交叉分析不是要做正式統計檢定，而是用來找出值得進一步歸納的設計線索。樣本小時不要切太細，例如「20–25 歲女性學生且使用 Instagram 且願意匿名」這樣會讓每格人數太少，沒有分析意義。

**進入下一節點門檻**

| 門檻                  |
| ------------------- |
| 至少完成 3 組交叉分析        |
| 每組交叉分析都有一句解讀        |
| 解讀能連到 Persona 或功能方向 |
| 沒有使用過度精準或過度推論的語氣    |

---

### 節點 3：開放題快速主題分析

這一步由 C 主責，A 負責把關，B 協助補數據連結。

NN/g 指出，主題分析是將質性資料拆解、組織並標記觀察與引述的系統性方法。你們可以做輕量版，但不能只挑幾句好看的回答。([Nielsen Norman Group][1])

**建議編碼方式**

每則開放題回答最多標 1–3 個代碼，例如：

| 回答內容       | 代碼          |
| ---------- | ----------- |
| 怕被認識的人看到   | 匿名需求、曝光顧慮   |
| 想聽有經驗的人怎麼做 | 經驗參考        |
| 不想被說教      | 低壓力互動、跨世代顧慮 |
| 怕詐騙或個資外洩   | 安全信任        |
| 不知道怎麼開始聊天  | 互動引導        |
| 願意分享但怕麻煩   | 分享者界線       |

**建議主題**

| 主題     | 可能意義                |
| ------ | ------------------- |
| 匿名與安全感 | 使用者想表達真實困擾，但怕曝光     |
| 經驗參考需求 | 使用者想聽過來人或類似情境者的經驗   |
| 低壓力互動  | 使用者怕被批評、被說教、尷尬      |
| 信任與風險  | 使用者擔心詐騙、騷擾、個資       |
| 分享者界線  | 願意分享者需要保護與互動界線      |
| 操作簡單   | 對 App 或數位平台有門檻者需要引導 |

**進入下一節點門檻**

| 門檻                      |
| ----------------------- |
| 所有有效開放題都已閱讀             |
| 每則重要回答至少有 1 個代碼         |
| 已整理出 5–8 個主題            |
| 每個核心主題至少有 2–3 則回答或選項題支持 |
| 每個核心主題有 1 句代表語句或摘要      |

---

### 節點 4：親和圖分群

這一步三人一起做，不建議一個人獨立完成。

**做法**

把以下內容做成便利貼：

| 來源   | 便利貼內容              |
| ---- | ------------------ |
| 描述統計 | 「不少填答者重視匿名」        |
| 交叉分析 | 「某些煩惱類型更需要經驗參考」    |
| 開放題  | 「怕被熟人知道」           |
| 開放題  | 「想聽過來人怎麼做」         |
| 團隊觀察 | 「匿名功能若沒有安全機制可能有風險」 |

然後分成 5–7 群，並寫成洞察句。

**洞察句格式**

> 使用者需要【需求】，因為【原因／痛點】。

例如：

> 使用者需要低曝光的表達空間，因為現實人際關係會讓他們擔心被認出或被評價。

**進入 Day 2 前門檻**

| 門檻                    |
| --------------------- |
| 已整理 5–7 個需求群組         |
| 每個群組都有洞察句             |
| 每個群組都有問卷或開放題依據        |
| 已初步判斷哪些群組可能形成 Persona |
| 已列出至少 3 個待外部資料佐證的問題   |

## Day 2 各節點執行重點與門檻

### 節點 5：桌面研究與競品對照

因為你們不做訪談，這一步很重要。它不是拿來「證明你們一定對」，而是用來補強與檢查風險。

NN/g 對 triangulation 的定義是使用多種資料來源或分析方法來提升研究可信度。你們雖然沒有訪談，但仍可用「問卷統計 + 開放題主題 + 桌面研究 + 競品對照」做低成本三角檢核。([Nielsen Norman Group][8])

**每人查什麼**

| 成員 | 查找方向                          | 產出        |
| -- | ----------------------------- | --------- |
| A  | Persona、proto-persona、JTBD 方法 | 方法佐證      |
| B  | 高齡友善、問卷小樣本限制、UX 可用性原則         | 設計限制與注意事項 |
| C  | 匿名社群、問答平台、經驗分享平台、安全機制         | 競品對照矩陣    |

**建議競品／類似平台 3 個即可**

| 類型    | 可觀察平台                        |
| ----- | ---------------------------- |
| 匿名交流  | Dcard 匿名、Reddit、匿名問答社群       |
| 經驗分享  | Dcard 經驗文、Medium、方格子、PTT     |
| 問答／互助 | Quora、知乎、Stack Exchange 類型平台 |

**競品對照欄位**

| 檢查項目     | 觀察重點           |
| -------- | -------------- |
| 是否允許匿名   | 匿名程度、暱稱、身份揭露方式 |
| 是否有主題分類  | 使用者如何找內容       |
| 是否有檢舉／封鎖 | 如何處理騷擾或不當內容    |
| 是否有互動規範  | 是否提醒使用者友善回覆    |
| 是否有新手引導  | 初次使用是否容易理解     |
| 是否有信任機制  | 隱私、安全、內容管理如何呈現 |

**進入下一節點門檻**

| 門檻                    |
| --------------------- |
| 至少完成 3 個外部可信來源整理      |
| 至少完成 3 個競品／類似平台觀察     |
| 每個競品至少檢查 5–6 個欄位      |
| 已標示哪些問卷洞察被外部資料支持      |
| 已列出安全、隱私、互動品質、操作負擔等風險 |

### 節點 6：建立洞察主題與需求假設

這一步由 A 主責，B、C 補證據。

**格式建議**

| 洞察主題      | 需求假設                    | 證據來源          | 信心等級 | 待驗證問題            |
| --------- | ----------------------- | ------------- | ---- | ---------------- |
| 低壓力安全表達   | 有一類使用者想匿名說出煩惱，但擔心曝光或被評價 | 問卷、開放題、匿名平台觀察 | 中高   | 匿名是否真的提高發問意願？    |
| 經驗型決策支持   | 有一類使用者面對人生選擇時想參考過來人經驗   | 問卷、開放題、經驗分享平台 | 中高   | 使用者偏好故事、建議還是一對一？ |
| 有界線的經驗分享  | 有一類使用者願意分享，但需要互動界線      | 問卷、開放題、社群規範觀察 | 中    | 分享者需要哪些保護機制？     |
| 謹慎觀望與信任確認 | 有一類使用者會先確認安全與隱私才互動      | 問卷、開放題、競品安全機制 | 中高   | 哪些安全說明最能降低疑慮？    |

**注意事項**

沒有訪談時，請避免寫：

> 使用者一定會……

建議寫：

> 目前資料推測……

或：

> 本階段初步判斷……

**進入下一節點門檻**

| 門檻                   |
| -------------------- |
| 已形成 3–5 個洞察主題        |
| 每個洞察至少有 2 種資料來源支持    |
| 每個洞察都有信心等級           |
| 每個洞察都有待驗證問題          |
| 能收斂成 2–3 個主要 Persona |

### 節點 7：產出 2–3 個假設型 Persona

2 天內建議產出 **2–3 張**，不要做太多。

**最適合你們專題的 3 張 Persona**

| Persona  | 核心描述                          | 對應洞察             |
| -------- | ----------------------------- | ---------------- |
| 匿名傾訴者    | 遇到煩惱想找出口，但不想被熟人知道或被評價         | 匿名、安全感、低壓力互動     |
| 經驗尋求者    | 面對職涯、感情、家庭或人生選擇時，想聽有類似經驗的人怎麼做 | 經驗參考、跨人生階段交流     |
| 謹慎分享／觀望者 | 願意使用或分享，但會先確認安全、界線與操作是否清楚     | 信任、安全、分享者界線、操作負擔 |

如果資料中「分享者」與「謹慎觀望者」差異明顯，可以拆成兩張；如果時間不足，可以合併成「謹慎參與者」。

**Persona 卡片欄位**

| 欄位         | 內容                      |
| ---------- | ----------------------- |
| Persona 名稱 | 匿名傾訴者                   |
| 一句話描述      | 想說出煩惱，但需要安全、匿名、低壓力的環境   |
| 代表情境       | 遇到職涯、感情、家庭或情緒壓力，不方便找熟人說 |
| 核心需求       | 安心表達、被理解、獲得經驗或支持        |
| 主要痛點       | 怕被認出、怕被評價、怕個資外洩         |
| 目前替代方案     | 問朋友、看社群、搜尋文章、自己消化       |
| 產品機會       | 匿名發問、主題分類、友善回覆提醒、隱私說明   |
| 證據來源       | 問卷題目、開放題主題、競品觀察         |
| 信心等級       | 中高／中／低                  |
| 待驗證假設      | 使用者是否真的願意匿名發問？          |

**進入下一節點門檻**

| 門檻                           |
| ---------------------------- |
| 完成 2–3 張 Persona             |
| 每張 Persona 至少有 2–3 個證據來源     |
| 每張 Persona 有明確情境、需求、痛點       |
| 每張 Persona 能導出功能或 User Story |
| 每張 Persona 有信心等級與待驗證假設       |

---

### 節點 8：JTBD、User Stories 與 Acceptance Criteria

這一步由 C 主筆，A 檢查是否符合 Persona，B 補數據來源。

**JTBD 格式**

> 當我遇到【情境】時，我想要【完成的事】，讓我可以【得到的結果】。

**User Story 格式**

> 作為一位【Persona】，我希望【功能／行為】，以便【價值】。

**範例**

| Persona | JTBD                               | User Story                             |
| ------- | ---------------------------------- | -------------------------------------- |
| 匿名傾訴者   | 當我遇到不方便告訴熟人的煩惱時，我想匿名說出來，讓我可以降低心理壓力 | 作為一位匿名傾訴者，我希望可以匿名發問，以便在不被熟人知道的情況下表達困擾  |
| 經驗尋求者   | 當我面臨人生選擇時，我想看到有類似經驗的人怎麼做，讓我獲得參考方向  | 作為一位經驗尋求者，我希望依主題瀏覽經驗分享，以便找到和自己情境接近的內容  |
| 謹慎參與者   | 當我第一次接觸平台時，我想先確認是否安全，讓我決定是否願意互動    | 作為一位謹慎參與者，我希望在使用前看到清楚的隱私與安全說明，以便安心開始使用 |

**核心 User Stories 建議 6–8 則即可**

| 優先級    | User Story |
| ------ | ---------- |
| Must   | 匿名發問       |
| Must   | 主題分類瀏覽     |
| Must   | 隱私與安全說明    |
| Must   | 檢舉／封鎖不當互動  |
| Should | 經驗故事卡      |
| Should | 回覆前友善提醒    |
| Should | 新手引導       |
| Could  | 配對推薦或個人化推薦 |

**Acceptance Criteria 範例**

| User Story | Acceptance Criteria                 |
| ---------- | ----------------------------------- |
| 匿名發問       | 使用者可選擇匿名；發問前看到匿名說明；可選主題；送出後可刪除或隱藏   |
| 主題分類瀏覽     | 使用者可依職涯、家庭、感情、情緒等主題瀏覽；每則內容有標題、摘要與標籤 |
| 隱私與安全說明    | 註冊或發問前可看到簡短隱私摘要；說明資料如何顯示、如何檢舉與封鎖    |
| 檢舉／封鎖      | 每則回覆旁有檢舉入口；使用者可封鎖特定對象；檢舉後有狀態提示      |

**進入下一節點門檻**

| 門檻                                    |
| ------------------------------------- |
| 至少 6 則 User Stories                   |
| 每張 Persona 至少對應 2 則 User Stories      |
| 每則核心 User Story 有 Acceptance Criteria |
| 每則 User Story 能追溯到 Persona 與問卷洞察      |
| 已初步分成 Must / Should / Could           |

### 節點 9：MVP 排序與風險標註

最後一定要做這一步，因為你們沒有訪談驗證，所以要用「風險標註」保護研究可信度。

**MVP 排序原則**

| 優先級            | 判斷方式                      |
| -------------- | ------------------------- |
| Must-have      | 多個 Persona 需要，且直接支撐平台核心價值 |
| Should-have    | 對某些 Persona 很重要，但第一版可簡化   |
| Could-have     | 有幫助但證據較弱或開發成本較高           |
| Won’t-have now | 證據不足、風險高或偏離核心目標           |

**建議 Must-have**

| 功能      | 理由              |
| ------- | --------------- |
| 匿名發問    | 回應低壓力表達與隱私需求    |
| 主題分類    | 回應經驗尋求與快速找到相關內容 |
| 經驗分享內容卡 | 回應「聽過來人經驗」需求    |
| 隱私與安全說明 | 降低謹慎使用者疑慮       |
| 檢舉／封鎖   | 匿名交流平台的基本安全機制   |

**風險標註表**

| 假設          | 風險            | 信心等級 | 後續驗證方式         |
| ----------- | ------------- | ---- | -------------- |
| 匿名會提高發問意願   | 使用者可能仍不信任匿名平台 | 中    | 後續概念測試         |
| 跨世代交流有吸引力   | 使用者可能怕被說教或尷尬  | 中    | 後續訪談           |
| 經驗分享比聊天更有價值 | 使用者可能更想即時互動   | 中    | Wireframe 測試   |
| 長者可接受平台操作   | 實際操作可能卡關      | 中低   | Usability test |
| 分享者願意回覆     | 分享者可能擔心責任或壓力  | 中    | 分享者訪談          |

**完成門檻**

| 門檻                                      |
| --------------------------------------- |
| Must-have 控制在 5–7 項                     |
| 每個 Must-have 都能追溯到 Persona 或 User Story |
| 每個高風險假設都有待驗證方式                          |
| 報告中明確寫出「未進行訪談，因此 Persona 為假設型」          |
| 團隊能用產出直接進入 User Flow 或 Wireframe        |

# 最推薦的 3 人協作方式

## A：研究整合與品質把關

適合擔任類似 PM / UX Research Lead 的角色。

**負責**

| 工作           |
| ------------ |
| 控制研究問題與流程    |
| 主持親和圖與洞察收斂   |
| 寫 Persona 架構 |
| 確認每個結論有證據    |
| 標示研究限制與待驗證假設 |

**最重要任務**

A 要避免團隊過度推論。例如不能把「本次填答者中較多人在意匿名」寫成「所有使用者都需要匿名」。

---

## B：資料整理與量化分析

適合負責試算表、圖表、數據摘要。

**負責**

| 工作      |
| ------- |
| 描述統計    |
| 多選題整理   |
| 3 組交叉分析 |
| 製作圖表    |
| 建立資料追溯表 |

**最重要任務**

B 要確保每一個 Persona、User Story、MVP 功能都能追溯到資料來源。

建議做一張「證據追溯表」：

| 產出     | 對應資料                  |
| ------ | --------------------- |
| 匿名傾訴者  | 匿名意願題、隱私顧慮題、開放題「怕被認出」 |
| 經驗尋求者  | 想獲得幫助類型題、開放題「想聽過來人經驗」 |
| 隱私安全說明 | 隱私顧慮題、競品安全機制          |
| 主題分類   | 煩惱類型題、競品分類方式          |

---

## C：開放題、競品與產品轉換

適合負責文字分析、競品觀察、User Stories。

**負責**

| 工作                  |
| ------------------- |
| 開放題編碼               |
| 主題命名                |
| 競品矩陣                |
| JTBD                |
| User Stories        |
| Acceptance Criteria |

**最重要任務**

C 要把「研究洞察」轉成「產品可以做的事情」，但不能跳太快直接寫功能。要先寫 JTBD，再寫 User Story，最後才寫功能。

# 最終建議

在你們目前條件下，最適合的 2 天工作策略是：

> **Day 1 專注把問卷變成洞察，Day 2 專注把洞察變成 Persona、User Stories 與 MVP。**

不要把時間平均分散到太多方法。最重要的是守住三個品質底線：

1. **每個 Persona 都要能追溯到問卷或開放題資料。**
2. **每個 User Story 都要能對應 Persona 與 JTBD。**
3. **所有未經訪談驗證的結論，都要標示為假設與待驗證。**

這樣即使只有 2 天、3 人小組，也能做出品質不至於流失太多、且能被報告與產品設計採用的研究成果。
