# Appendix_使用情境格式比較

## 目的

本文件用於比較 `03_假設型 Persona初稿.md` 中新增「真實使用情境」欄位時可採用的兩種格式，協助後續人工定稿前決定欄位寫法。

目前仍停留在 03 Persona 初稿階段，不進入 `04_User Stories`。

## 選項 1：使用前／使用中／使用後

### 格式範例

| 欄位 | 內容 |
|---|---|
| 真實使用情境 | 使用前：小安正在考慮轉職，但不想讓同事知道，也覺得朋友沒有類似經驗。他先看平台的匿名、安全與信箱說明，確認不是公開發文後，才願意嘗試。<br>使用中：他選擇「職涯選擇」主題，填寫自己的困惑，設定希望收到「類似經驗分享」而不是直接建議，並送出一對一匿名信箱。<br>使用後：他願意等待一段時間，但需要看到等待狀態。如果沒有人回覆，他希望平台提供相關範例內容或替代建議，而不是讓問題消失。 |

### 優點

- 非 UI/UX 背景的人容易看懂，因為時間順序清楚。
- 很容易轉成 User Stories。
- 能明確看出產品要支援哪些流程。
- 能避免 Persona 變成功能清單。
- 對 P-01/P-02 都適用，尤其能補出「等待」「拒絕」「略過」「回覆後責任」這些重要狀態。

### 缺點

- 表格會變長。
- 寫起來比較像流程分析，不像傳統 Persona 卡片那麼簡潔。
- 如果每段寫太細，可能會提前進入 User Flow 或 User Stories。

### 適合現在的原因

- 團隊接下來就是要往 User Stories 走。
- 團隊非 UI/UX 背景，需要清楚知道「使用者在什麼時候需要什麼」。
- 目前 03 的問題正是「情境可能不夠支撐後續故事」。

## 選項 2：單段情境敘述

### 格式範例

| 欄位 | 內容 |
|---|---|
| 真實使用情境 | 小安正在考慮轉職，但不想讓同事知道，也覺得朋友沒有類似經驗。他看到平台主打一對一匿名經驗信箱後，會先確認匿名與安全說明，再選擇職涯主題、描述自己的困惑，並希望收到類似經驗分享而不是直接建議。送出後，他願意等待回覆，但如果一直沒有人回，他會希望平台提供等待狀態、相關範例內容或替代建議。 |

### 優點

- 讀起來比較自然，像 Persona 情境故事。
- 文件比較精簡。
- 適合放在報告或簡報裡，閱讀負擔小。
- 比較不會讓 Persona 看起來像流程規格。

### 缺點

- 後續拆 User Stories 時，需要再人工整理一次。
- 使用前、使用中、使用後可能混在一起，容易漏掉等待、拒絕、回覆後責任等細節。
- 團隊討論時比較難逐段檢查「這段支撐哪個功能或故事」。

### 適合的情況

- 如果 03 只是要給老師或組員快速理解 Persona。
- 如果後續 User Stories 會另外完整拆流程。
- 如果團隊想讓文件讀起來更像研究報告，而不是分析表。

## 初步建議

建議採用 **使用前／使用中／使用後**。

理由是目前 03 的核心任務不是做漂亮 Persona，而是讓非 UI/UX 背景團隊能順利往 `04_User Stories` 走。分成三段後，每段幾乎可以直接對應下一步：

| 階段 | 後續可支撐方向 |
|---|---|
| 使用前 | 導覽、安全說明、差異化說明 |
| 使用中 | 匿名投遞、主題選擇、配對條件、回覆規範 |
| 使用後 | 等待狀態、替代內容、拒絕/略過、檢舉/結束互動 |

若擔心文件太長，可折衷：新增欄位名稱為 `真實使用情境`，內容用三個短句標示 `使用前 / 使用中 / 使用後`，每段 1-2 句即可。
